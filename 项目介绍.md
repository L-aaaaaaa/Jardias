# Jardias项目介绍

## 项目概述

本项目并非生产级项目，而是一套智能体系统理论的参考实现及构造性证明。当前Jardias实现是一个让 AI 不只是回答问题，而是自主协作、记忆成长、时间感知的认知主体框架，最大程度做好底层抽象，为将来的系统进化提供开放接口。

## 项目名称

Just A Rather Dimension-Free-Updating Intelligent Actor System ，Jardias
佳递叶思，佳-好的，递-递归，叶思-贝叶斯更新，Actor是智能体/行动者/演员。

JARDIAS is just a rather dimension-free-updating intelligent actor system. It doesn't fly, it doesn't fight. But it keeps updating to break the dimensional wall.

《Dimension-Free Updatism》（维度自由更新论）是项目作者提出的一种范式革命的可操作工程框架，核心思想是当参数更新触碰到次元壁，可以通过策略升维来实现破壁操作。



## 命名体系

根据《命名即架构》（链接），本项目中执行以下命名重构方案

AI模型（AI Model）→智能基元（IPU）
模型调用管理模块→义脑（Yinao）
矢量文本（token）→矢量化的文本保持token，计量单位为智点（ICP）
图像的矢量像素（Pixel Patch）→同样保留Pixel Patch，计量单位为智点
智能体（Agent）→智能体/智能演员（Actor）
智能体系统（Agent System）→智能体/智能演员系统（Actor System）
扮演具体设定的智能体→角色（character），使用时直接称呼具体角色名

# 理论支撑

《HEEL》：以「可注入性」作为正交分类判据的记忆分层方案——只问数据"能不能直接用"，不问"是什么"，由此推出 History / Experience / External / Logic 四层。

《时策论文》：把"自然语言时间意图必须预编译为结构化规则"这一假设推翻——LLM 本身是调度系统的决策核心，调度器只负责"等-触发"，错过补偿、模式切换、动态取消都由 LLM 在运行时语义推导。

《配置参数化》：把模型名、温度值、角色描述、工具列表等配置从外部 YAML 迁移到 LLM 可读写的记忆层——配置即记忆，Agent 由此获得自指涉的操作平面，涌现出智能路由、多模态编排、自组织角色生态与自触发深度精炼四个能力。

《Agent长程任务执行架构方案》：提出"成熟的记忆方案应该实现上下文占用的永久宽裕"这一架构正确性判据，将 OS 内存管理思想映射到 Agent 上下文管理，给出长程任务执行的工程骨架。

《长程任务与关注域》：以「关注域」作为与可注入性正交的第二条认知资源管理原则——可注入性问"能不能进"（资格），关注域问"此刻该不该进"（激活），二者序贯决策；配套的状态文件机制将目标锚定 $Q$ 与工作集 $W_Q$ 工程实例化。

《AgentOS范式总纲》：提出"破壁定理"作为整套架构的元原理——单一标量管理多个正交维度时准确率上限 50%，唯一突破路径是策略升维；将 HEEL、就绪性-触发度分离、配置即记忆、角色-分身-义脑解耦依次作为第一至第四次升维。

《AIOS》：将 Agent 视为"认知操作系统"，论证操作系统设计哲学（进程、虚拟内存、页表、文件系统等）整套迁移到 Agent 架构的可行性与必然性，给出与主流"零件市场"思路的本质差异。

《Agent OS范式下的研究议程》：以"基础深化→系统协同→社会生态→度量评估"四层组织 Agent OS 范式必然引出但尚未解决的问题——把注意力预算、负向记忆、自适应遗忘率等列为开放研究课题，邀请跨学科研究者共同参与。

《Agent Context Protocol》：针对当前 LLM API 用扁平 `messages` 数组承载一切信息的局限，提出 ACP 协议草案——为会话状态、时间戳、分层记忆等结构化信号设计专用载体，解放"在字符串内嵌入约定格式"的工程妥协。

# 附录：功能对照表

Jardias 的功能边界由"哪些能力是别的项目从架构前提上做不到的"来界定。下表把四大特色能力逐项拆解到"实现机制 + 同类项目对照"两层：

| 能力 | 实现机制 | LangChain / CrewAI / AutoGen 是否能做到 |
|---|---|---|
| **智能体创建角色辩论（自触发深度精炼）** | `create_character` 工具让 Agent 在推理中动态创建并配置新角色（指定 provider / ipu / temperature / top_p / thinking），多个角色共享 history 流并行推理 | 不能——多 Agent 角色需用户在框架外预先定义并填配置；Agent 自身没有"创建 Agent"的元操作 |
| **双角色自我手术（递归自主进化）** | `send_to_character` + `update_runtime` 工具链 + `update_identity` 让创建出的角色反过来承担后续对话，并可通过工具调用修改自己或对方的运行时配置 | 部分——能创建多 Agent，但配置（模型/温度/工具集）属于框架外部，Agent 不能读写自己的配置 |
| **上下文控制（HEEL 记忆架构 + ACP）** | O(1) 4 段消息上下文 + L1/L2/L3 摘要 + 关注域判据 + 状态文件锚定目标 $Q$——历史压缩按"加工状态"而非"语义类型"分类 | 不能——主流方案仍按消息序列线性增长上下文，或只做摘要压缩；没有"就绪性-触发度"分离的双层决策 |
| **时策系统（语义驱动调度）** | `shice_schedule_add/list/cancel` 工具 + `TemporalScheduler` 时间戳队列模型——LLM 在运行时推理下一个绝对时间戳并预计算所有触发点，错过由 LLM 语义推导补偿与模式切换 | 不能——主流调度（Cron / Airflow / Temporal）依赖规则预编译；自然语言意图必须翻译为结构化表达式，意图保真度不可逆地丢失 |
| **歧义/边界确认** | system prompt 注入"用户描述时间若存在歧义/边界不清应主动询问"的硬规则 + LLM 语义识别 | 部分——能加 guardrail，但调度器本身仍要求用户输入精确规则 |
| **语义化错过补偿** | 调度器只管"过期≤60s 立即触发并标记 late_sec"，补偿策略（补哪些、改哪些、要不要切间隔）全部由 LLM 在 system_trigger 消息中语义推导 | 不能——只有预设策略枚举（SKIP / FIRE_ONCE / FIRE_ALL），无法做"重要的检查任务补上、播报类跳过"的语义区分 |
| **动态干预** | 用户的取消信号以普通消息注入 history 流，LLM 识别后调用 `shice_schedule_cancel` 一次性撤销 | 部分——能监听用户输入但需要额外的取消事件总线；不与调度器语义融合 |
| **跨 Session 精确回忆** | L1 段摘要 + O(1) 4 段上下文 + 角色绑定终身记忆库（`character_data/<timestamp>-<name>/`）；Session 只是视图，身份不与会话同生灭 | 部分——能持久化历史，但身份与 Session 通常强绑定；切换 Session 等于身份销毁 |
| **多模型热切换** | `update_runtime` 工具 + IPU_REGISTRY 路由 + `request_switch` 切换标志在工具执行间隙被检测；切换不重启 | 不能——主流方案需重启进程或重连客户端才能切换底层模型 |
| **`··` 快速路由** | `··角色名` / `··工具名` 语法绕过 LLM 决策，确定性执行跨角色切换或工具直调 | 不存在——主流方案没有把内部能力暴露为用户可寻址命令空间的设计 |
