# 贾维斯 vs 市面 Agent 框架：特性对照表

> 不是"做得更好"，是"公理体系不同"。以下每个差距都是旧范式架构前提的逻辑后果，不是工程水平差距。

---

## 一、总览：不是零件竞争，是范式跃迁

| 维度 | LangChain | CrewAI | AutoGen | MemGPT/Letta | 贾维斯（Agent OS） |
|---|---|---|---|---|---|
| **架构哲学** | 零件拼装（Memory + Tool + Chain） | 多 Agent 角色扮演 | 多 Agent 对话通信 | OS 隐喻（仅限记忆层） | 完整认知操作系统，破壁定理统一 |
| **记忆管理判据** | 语义类型（Buffer/Summary/Vector） | 语义标签 + 检索评分 | 语义类型 | 语义类型 + 页表（以 Session ID 为键） | 就绪性 ℐ(d)（正交于语义，只问加工状态） |
| **记忆跨会话持久化** | Session 内，需手动搬运 | Session 内，需手动搬运 | Session 内，需手动搬运 | Session 内，需手动搬运 | 终身记忆库 M，不依赖 Session |
| **身份管理** | 提示词 = 身份 | 人工定义每个 Agent | 人工定义每个 Agent | 提示词 = 身份 | 角色-分身-义脑三层解耦，换模型 ≠ 换人格 |
| **配置管理** | 外部 YAML/JSON，不可运行时修改 | 外部配置，Agent 不可见 | 外部配置，Agent 不可见 | 外部配置 | 配置即记忆，Agent 自主读写 |
| **调度范式** | Cron / APScheduler 外包 | Cron / APScheduler 外包 | Cron / APScheduler 外包 | 无原生调度 | 时策：存自然语言指令，LLM 运行时推理 |
| **多 Agent 创建方式** | 人工编写 | 人工定义角色 + 任务 | 人工定义角色 + 会话 | 无多 Agent | Agent 自主提案创建对口角色（含模型、温度、工具集） |

---

## 二、逐项差距详析

### 2.1 记忆管理：就绪性 vs 语义分类

| | 所有基于语义分类的方案 | 贾维斯（HEEL） |
|---|---|---|
| **判据** | "这是什么类型的数据"→ 决定怎么管 | "数据处于什么加工状态"→ 决定在哪层 |
| **典型案例** | 一段包含过敏信息的对话 = "对话历史"= 跟 Session 同生灭 | 原始对话在 History，提炼后的 `{"fact":"青霉素过敏"}` 在 Experience |
| **后果** | 语义类型相同的两种数据（一条闲聊 + 一条含关键偏好的对话），管理策略无差别 | 加工状态不同的数据自动分层，关键信息不会随历史滚动被丢 |
| **前瞻兼容** | 上下文窗口扩大 → 改截断阈值，代码要改 | 上下文窗口扩大 → 调高 Experience 的 max_tokens，架构不动 |

**做不到的能力**：语义分类方案无法区分"一段闲聊"和"一段包含关键偏好的对话"——它们戴着同样的"对话历史"标签，被同样的截断策略处理。聊了 1 小时，前面的关键偏好可能已被挤出窗口，而 30 分钟前的寒暄还在。

### 2.2 上下文构建：就绪性-触发度分离 vs 复合评分

| | LangChain / CrewAI / MemGPT | 贾维斯 |
|---|---|---|
| **机制** | 单一标量评分（语义相似度 + 时效性 + 重要性加权求和）| 就绪性审查（资格）→ 触发度排序（激活）两层序贯 |
| **核心 bug** | 无法区分一条数据的高分来自"格式好、语义自足"还是"恰好跟当前查询文本相似" | 资格与激活分开，颁奖者不再同时是审查者 |
| **后果** | 已就绪但无关的数据抢占上下文窗口（如昨天的摘要跟今天的查询语义相似但任务无关） | 先过滤掉未就绪数据，再在候选集中基于任务目标激活 |

**做不到的能力**：复合评分方案在长程任务中必然出现"已就绪但无关"数据挤占窗口的问题。不是调参能解决的——是策略降维的结构性缺陷（破壁定理已严格证明）。

### 2.3 角色自主创建与管理

| | CrewAI / AutoGen | 贾维斯 |
|---|---|---|
| **多 Agent** | ✅ 支持 | ✅ 支持 |
| **角色定义** | 人工编写：定义名称、角色描述、工具集 | Agent 自主提案：分析任务特征 → 设计角色方案 → 调用工具创建 |
| **角色配置** | 人工预设，运行时不可变 | Agent 自主设定并可在运行时修改（模型、温度、工具集） |
| **权限/能力隔离** | 取决于开发者手动设计 | Agent 描述需求 → 系统自动分配，后续可自我调整 |
| **跨角色记忆** | 共享或独立，人控开关 | 角色绑定独立记忆库 M，分身共享角色记忆但运行态隔离 |
| **角色间交互** | 人工编排对话流程 | Agent 自主发起对方分析和配置优化 |

**做不到的能力**：

- **Agent 创建 Agent**：用户说"你创建一个擅长处理金融分析的角色来帮我看看这笔交易"，贾维斯能分析任务需求后自动创建对口角色（分配模型、设置温度、定义工具权限），CrewAI/AutoGen 需要人工先定义所有角色。
- **自组织角色生态**：当任务复杂度超出单个角色能力时，贾维斯自动提案多角色方案——"这个问题涉及法律、金融和技术三个维度，我建议创建三个分身分别审查，然后汇总意见"。

### 2.4 配置即记忆：Agent 的自指涉能力

| | 所有传统框架 | 贾维斯 |
|---|---|---|
| **配置位置** | 外部 YAML/JSON/代码，Agent 推理不可达 | Experience 层，LLM 可直接读写 |
| **模型选择** | 用户手动指定或固定路由规则 | Agent 读取模型能力映射表 + 评估任务复杂度 → 自主选模型 |
| **配置修改** | 需重启或重载，需人工介入 | Agent 自主修改并即时生效，修改持久化到记忆库 |
| **跨角色配置污染** | 全局配置，改一个影响所有 | 按角色隔离，角色 A 调高温度不影响角色 B 的科学推理 |

**做不到的能力**：

- **智能路由**：简单问题 Agent 自主用小模型省钱，复杂推理自动切大模型——传统框架需要人工写路由规则
- **自触发配置优化**：Agent 发现自己在某个领域回答质量下降 → 自主分析原因 → 调整温度或换模型 → 验证效果。例如日志中"双角色自我手术"场景：角色 1 创建角色 2，双方互相审查配置并提出修改建议，然后验证修改效果

### 2.5 时策：运行时推理 vs 规则预编译

| | Cron / APScheduler / Airflow（所有框架的后端） | 贾维斯（时策） |
|---|---|---|
| **持久化数据** | 结构化规则（Cron 表达式 / JSON / DAG） | 用户原始自然语言指令（不可变） |
| **决策机制** | 规则解析器：`规则 + 当前时间 → 触发/不触发` | LLM 运行时推理：`原始指令 + 执行历史 → 下一个绝对时间戳` |
| **复杂日历** | 需额外代码（农历、节假日、外部条件） | LLM 内置常识或调用工具，无需增加核心调度代码 |
| **调度代码复杂度** | O(指令类型数) — 每新增一类指令需新增解析代码 | O(1) — 始终是同一"等待-触发-询问"循环 |
| **错过补偿** | 预设枚举 SKIP / FIRE_ONCE / FIRE_ALL | LLM 根据指令深层语义动态推导补偿策略 |
| **任务分类** | 一次性 vs 循环，两套代码路径 | 不存在一次性任务——只有 LLM 决定不再继续的任务 |

**做不到的能力**：

- **"每周一三五 9 点，节假日顺延"** — Cron 只能做前一半，后一半需要外部代码；时策原生支持，LLM 每次决策时自己判断
- **"120 秒后每 5 秒说一个水果，错过 2 次就改 10 秒间隔"** — 传统系统需要人工建模条件切换；时策直接从自然语言推导（已在日志中验证）
- **"重要的检查任务必须补上，准点播报类错过就算了"** — 传统系统要么全补要么全跳；时策能理解语义差异做差异化补偿

## 三、"做不到"的根本原因：不是工程水平，是架构前提

| 框架 | 做不到的核心原因 |
|---|---|
| **LangChain** | Memory 三种模式（Buffer/Summary/Vector）都将持久化绑在 Session 生命周期上。记忆随 Session 生灭——无法实现 Agent 自主创建角色和配置（因为无跨 Session 记忆） |
| **CrewAI** | 角色需人工定义，Agent 只能"执行已定义好的角色"。评分检索机制的对象是语义整体——无法区分"已就绪"和"需加工"的数据 |
| **AutoGen** | 多 Agent 通信强大但无统一记忆和配置管理。角色切换需手动搬运人格设定，无法实现配置自指涉 |
| **MemGPT/Letta** | OS 隐喻最接近但只覆盖记忆层。页表以 Session ID 为键——跨会话页面共享不在设计范围内。无原生调度、无配置即记忆 |

**共同病灶**：这四个框架的架构前提从逻辑上阻止了贾维斯的能力，不是"可以改进去实现"的问题。就像在牛顿力学范式内，不管怎么优化公式都无法推导出质能方程——需要的不是更好的公式，是换一套公理。

## 五、不可做之事（诚实边界）

贾维斯也有明确的做不到：

- **语义幻觉放大**：四次升维都依赖 LLM 正确理解语义，如果 LLM 判断错了，偏差会逐级放大。这不叫 bug，叫"能力边界的同源性"
- **硬实时调度**：延迟 <10ms 的场景，时策不适用（LLM 推理延迟数百毫秒起步）
- **超高频调度**：每秒级触发，LLM 调用成本和延迟均不可行
- **产品级性能优化**：当前是架构验证，不是性能极致
