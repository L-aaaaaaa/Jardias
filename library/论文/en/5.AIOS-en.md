# AIOS Cognitive Kernel: Interface Specification and Iterative Framework (v1.0)

---

## Abstract

The current development of language-agent systems is trapped in a "parts market" predicament — memory, planning, scheduling, configuration, and multi-role collaboration each have their own independent solutions, but lack a unifying architectural philosophy. As a result, system complexity grows exponentially with the number of modules, context management inflates linearly, and inter-module coordination is fragile. Meanwhile, the AIOS (Artificial Intelligence Operating System) field has been jointly established by academia and industry — from Rutgers University's LLM Agent OS to the on-device practice of Honor MagicOS 10 — yet existing AIOS work generally focuses on resource scheduling and task orchestration, leaving two fundamental capabilities still blank: **cognitive resource management** (memory, attention, identity) and **meta-cognitive self-evolution**.

This paper proposes the architecture of the AIOS Cognitive Kernel — a unified design philosophy that maps the core abstractions of operating-system design onto agent architecture in an engineering manner. The kernel's meta-principle is a more fundamental principle of cognitive architecture — the **Wall-Breaking Principle**: any system that attempts to manage multiple orthogonal decision dimensions with a single scalar has a structurally insurmountable ceiling on its decision accuracy; the only path beyond this ceiling is to perform a Dimensional Elevation of Strategy — establishing an independent, sequentially executed decision layer for each orthogonal dimension.

The AIOS Cognitive Kernel carries the Wall-Breaking Principle through four key dimensions of cognitive architecture, each enactment being one operation of Dimensional Elevation of Strategy:

- **HEEL Cognitive Memory** (First Ascension): separates the semantic label of data from its processing state. The classification criterion is orthogonal to semantics — it asks only "can it be used?", not "what is it?". The agent thereby obtains a virtual cognitive memory space far larger than its context window.
- **Readiness–Activation Separation** (Second Ascension): separates qualification judgment from activation scheduling. The Readiness vetting layer is independent of the current goal and judges, solely on the data's intrinsic properties, whether the data meets the basic conditions for injection into the context; the activation management layer operates independently over the candidate set. The awarder is no longer simultaneously the inspector.
- **Configuration-as-Memory** (Third Ascension): elevates configuration management from external presets to internal memory. The agent gains a self-referential plane of operation — its reasoning can be directed not only at external tasks but also at its own mode of operation. This unlocks three meta-cognitive capabilities: intelligent routing, multimodal orchestration, and a self-organizing role ecology.
- **Character–Fenshen–Yinao Identity Decoupling** (Fourth Ascension): liberates identity continuity from transient sessions. A Character is a persistent persona with a lifelong memory repository; a Fenshen (分身, lit. 'avatar') is a temporary view of the same Character under a specific task; a Yinao (义脑, lit. 'prosthetic brain') is the module responsible for managing model invocation, hot-swappably replaceable. Sessions are demoted to views.

The core contribution of this paper is a **unification argument**: these four sub-frameworks are not independent inventions but successive unfoldings of the Wall-Breaking Principle across four distinct compressed dimensions of agent architecture. Each ascension identifies a previously compressed orthogonal dimension, builds an independent decision layer for it, and re-releases the suppressed information space. The four are interdependent — with any one missing, the system develops a specific structural vulnerability and cannot constitute a complete cognitive operating system.

Through the reference implementation "Jardias（佳递叶思）" we complete a constructive proof — this is not a paper blueprint but a runnable system. It demonstrates that the AIOS Cognitive Kernel is not only a self-consistent theoretical system but also an engineering-implementable architectural principle. Compared with existing AIOS frameworks, Jardias's core scheduling object is not tasks or APIs but cognitive resources — memory, attention, configuration, identity. This constitutes a paradigm leap for AIOS from a "task operating system" to a "cognitive operating system".

This paper positions itself as the Interface Specification v1.0 of the AIOS Cognitive Kernel — it defines the currently identified core modules and their boundaries, explicitly marking interfaces yet to be filled and ascension directions that can be anticipated. It is not a closed declaration of completion but an open invitation to iterate. Subsequent research can, following the unified extension principle defined herein (the Dimensional Elevation of Strategy operation), fill in new modules within the defined interface framework, or upgrade existing modules to higher versions.

The AIOS Cognitive Kernel is not positioned as "a better memory scheme" or "a better scheduling framework". It does not add a new stall to the parts market; it provides the whole market with a unified set of weights and measures and a set of traffic rules. The weights and measures are the Wall-Breaking Principle — it quantitatively proves that an undecoupled system is logically incomplete. The traffic rules are the sequential decision pipeline of Readiness vetting → activation scheduling — ask first "can it?", then "should it?". When all sub-frameworks cooperate under this set of rules, the agent is no longer "a chatbot that can call tools" but **a cognitive subject with the capacity for self-management**.

**Keywords**: AIOS Cognitive Kernel; Wall-Breaking Principle; Dimensional Elevation of Strategy; HEEL Cognitive Memory; Readiness–Activation Separation; Configuration-as-Memory; Character–Fenshen–Yinao decoupling; interface specification


## 1. Introduction

### 1.1 AIOS: An Emerging Field

AIOS (Artificial Intelligence Operating System) has been jointly established as an independent research direction by academia and industry. In late 2023, the foundational paper *LLM as OS, Agents as Apps* [10] proposed the vision of an AIOS–Agent ecosystem with "the large model as the kernel and agents as applications". In March 2024, the team of Zhang et al. at Rutgers University released the first engineered AIOS implementation — a three-layer architecture comprising a kernel (scheduling/memory/storage management), an SDK, and agent applications — achieving a 2.1× improvement in concurrent execution speed [11]. In January 2026, the journal *Computer Science* published a cover review systematically surveying the Agent OS field, listing AIOS as a canonical architecture and formally establishing its academic status as an independent research direction [12].

Industry is equally active. Honor MagicOS 10 completed the transition from a traditional OS to an AIOS; its YOYO agent, through a reinforcement-learning mechanism, achieved a leap in scenario coverage from 200+ scenarios to 3,000+ within three months. vivo open-sourced its Rust-based Blue River kernel; RT-Thread launched an edge-side AI solution. Major companies have published AIOS white papers one after another, positioning AIOS as infrastructure for "executable governance".

Yet although the field is established, consensus is far from formed. Everyone is building AIOS, but there is no unified standard for cognitive architecture — on two fundamental questions in particular, "how should cognitive resources be managed?" and "how does the system evolve itself?", existing work is almost entirely blank.

### 1.2 The Isomorphic Predicament in Industry

Industry's perception of AI architecture fragmentation is highly isomorphic to academia's diagnosis. In a year-long practice of AI-native work [9], Tencent Research Institute summarized a three-stage evolutionary path: from "widely using AI tools" to "precipitating personal capability assets" to "calling for methodological governance". Its practitioners wrote explicitly: "Skills keep being written and workflows keep being built, ever more numerous and messy, lacking a methodology to organize and govern them" — this is precisely the concrete industrial mirror of the "parts market predicament" diagnosed in this paper. The "trust gap" concept proposed by that report — "reliability does not mean never making mistakes; reliability means still being able to catch things after a mistake occurs" — aligns exactly with the design intent of the confidence markers in this paper: trust must be built upon knowable cognitive boundaries.

The report's "cognitive asset" concept essentially points to the structured precipitation of users' usage experience. In this paper's framework, the precise description of this process is: the user's raw interaction records (History layer) undergo Readiness vetting and processing, and are then upgraded into Experience units that can be directly injected into the context — exactly the data upgrade path defined by HEEL Cognitive Memory. This paper defines this path as the **Readiness processing chain of usage experience**.

This practical summary from industry corroborates that the diagnosis of this paper is not academic speculation but an architectural growing pain the entire AI industry is undergoing together. The difference: that report names the "pain"; this paper provides the underlying architectural cure.

### 1.3 The "Parts Market" Predicament in Current Agent Development

While the AIOS field develops rapidly, a fundamental architectural problem has long been overlooked.

**Status quo**: multiple memory schemes (LangChain Memory, MemGPT, Letta), planning schemes (ReAct, Plan-and-Solve, Tree-of-Thought), scheduling schemes (Cron, APScheduler), configuration schemes (YAML, JSON, UI forms), and multi-role schemes (AutoGen, CrewAI, CAMEL) exist. But there is no unified architectural philosophy among them. What developers face is a "parts market" — every part works, but you have to guess the assembly logic yourself.

**Consequences**: memory schemes are unaware of scheduling strategies — historical data generated by scheduled tasks never enters the memory hierarchy. Configuration schemes and role management are mutually unaware — changing model temperature requires a restart; switching roles requires reloading configuration. Context management is a "turf war" in which each module acts on its own — the planning module injects step lists, the memory module injects historical summaries, the retrieval module injects external knowledge; the three do not coordinate, and the context window becomes a tragedy of the commons. System complexity grows exponentially with the number of modules — each added module may produce unexpected interactions with all existing modules.

This is not a matter of some module not being good enough; it is the absence of an overarching architectural paradigm. What AIOS needs is not more and better parts, but an operating-system kernel that tells these parts how to cooperate.

### 1.4 Positioning of This Paper: An Interface Specification, Not a Declaration of Completion

This paper does not claim to have exhausted all modules of the AIOS Cognitive Kernel. On the contrary, it acknowledges that the current work is the **first phase** of a larger research agenda — identifying and decoupling the most core cognitive dimensions, and establishing a unified interface standard for subsequent extension.

Specifically, this paper accomplishes the following:
- **Defines the four core modules of the cognitive kernel**: Cognitive Memory (HEEL), Cognitive Scheduling (Readiness–Activation Separation), Cognitive File System (Configuration-as-Memory), Cognitive Process Isolation (Character–Fenshen–Yinao decoupling)
- **Defines the sequential interfaces between modules**: Readiness vetting → activation scheduling → configuration decision → identity routing
- **Defines the unified extension principle**: any new module must follow the Dimensional Elevation of Strategy operation — identify a compressed orthogonal dimension and establish an independent sequential decision layer for it
- **Defines the validation boundary of the reference implementation**: Jardias validates the engineering feasibility of the four core modules but does not cover the following to-be-filled sections

Content explicitly left open in this paper includes but is not limited to:
- **Security and permission model**: the "kernel mode / user mode" permission architecture of a cognitive operating system has not yet been formalized
- **Multi-instance consensus protocol**: the cognitive resource exchange protocol among multiple AIOS instances (ARIES) is still at the concept stage
- **Automated detection of cognitive drift**: mechanisms for detecting and correcting configuration drift remain to be systematized
- **Cross-Yinao continuity metrics**: metrics for persona stability when replacing the underlying model remain to be established
- **Higher-order cognitive dimensions**: affective modeling, formalization of value alignment, optimization of cognitive rhythms, etc., are candidate directions for future ascensions

This statement is not an admission of weakness but a definition of the rules of the game — this paper establishes an extensible architectural language in which any subsequent research can find its own interface position.

What the AIOS Cognitive Kernel does is to perform the same ascension operations in the domain of cognitive resource management — each ascension corresponds to an architectural separation already proven effective in the OS domain.

### 1.6 The Meta-Principle: The Wall-Breaking Principle

The theoretical soul of the AIOS Cognitive Kernel is a principle of cognitive architecture named the **Wall-Breaking Principle**.

First rigorously proven in the domain of agent memory management, its scope of applicability extends far beyond. Its core claim:

> **Any system that attempts to manage multiple orthogonal decision dimensions with a single strategy has a structurally insurmountable ceiling on its decision accuracy.** If and only if two orthogonal dimensions are completely conflated into the same scalar function, under conditions where a switch of environmental state requires reversing the decision priority order, the system's discrimination accuracy degenerates to the level of random guessing. In practical systems, if the conflation is incomplete — i.e., faint distinguishability of the dimensions remains in the scalar function — accuracy may lie anywhere between the random level and the theoretical optimum, but an **ineliminable structural error rate necessarily exists**, and this error rate increases monotonically as the degree of conflation deepens. This is because, after dimensional conflation, the system cannot tell whether a datum's high score comes from its advantage on dimension A or from its advantage on dimension B — the information of the two dimensions has been irreversibly conflated.

The only path beyond this ceiling is **Dimensional Elevation of Strategy** — re-expanding the compressed dimensions and establishing an independent, sequentially executed decision layer for each dimension. Moreover, Dimensional Elevation of Strategy is naturally recursive: whenever the system faces a new compressed orthogonal dimension, the ascension operation can be applied again.

The four sub-frameworks of the AIOS Cognitive Kernel are precisely the successive unfolding of the Dimensional Elevation of Strategy operation across four key dimensions of AIOS architecture. Each ascension identifies a previously compressed orthogonal dimension, builds an independent decision layer for it, and re-releases the suppressed information space.

## 3. The Meta-Principle of the AIOS Cognitive Kernel: The Wall-Breaking Principle

Any system that attempts to manage cognitive resources faces a fundamental architectural choice — decouple or conflate? The Wall-Breaking Principle gives the mathematical consequences of this choice.

### 3.1 The Dimensional Conflation Trap

In any system that must manage limited cognitive resources, two logically orthogonal decision dimensions naturally exist: (1) the intrinsic properties of data — whether it "can" be consumed (Readiness); (2) the relational properties of data with respect to the goal — whether it "should" be activated at this moment (activation propensity). These two dimensions have independent frames of reference, different update frequencies, and judgment logic that does not determine one another.

When a system attempts to use a single scalar function to proxy the decisions of both dimensions simultaneously, **dimensional conflation** occurs — the information of the two dimensions is irreversibly conflated into the same scalar space. (This condition was historically termed *strategic dimensional reduction*; the terminology here follows the final version of [2], Appendix C.) The system can no longer tell whether a datum's high score comes from its high Readiness (it is cleanly formatted and semantically self-contained) or from its high activation propensity (it happens to be textually similar to the current query).

This conflation is not the cost of approximation, not an engineering trade-off, not a performance bottleneck that can be optimized away by tuning. It is a structural trap proven, logically, to necessarily degenerate. We rigorously prove: **for any system in a state of complete dimensional conflation, the optimal accuracy in discriminating these two kinds of data has an exact and insurmountable upper bound — 50%** — precisely the level of random guessing. This is the **dimensional conflation trap**. This structural boundary — caused by dimensional compression and unbreakable within the existing space — we call the **dimensional wall**.

When the conflation is incomplete — faint distinguishability of the dimensions remains in the scalar function — accuracy may lie between 50% and the theoretical optimum, but an ineliminable structural error rate necessarily exists, and it increases monotonically with the degree of conflation, up to 50%.

### 3.2 Dimensional Elevation of Strategy: The Only Operation That Breaks the Trap

The proof of the dimensional conflation trap also points to the direction of breaking it. Since the problem lies in compressing two orthogonal dimensions into the same scalar function, the solution must be to **re-expand the compressed dimensions** — establish an independent decision layer for each dimension, and determine the sequential relation between them.

This operation is named **Dimensional Elevation of Strategy**. In concrete architecture it manifests as:

- **Readiness vetting layer** (qualification judgment): independent of the current goal, judges solely on the data's intrinsic properties whether the data meets the basic conditions for injection into the context. Data that fails review enters the processing pipeline rather than being injected directly.
- **Activation management layer** (activation judgment): within the candidate set that has passed Readiness vetting, judges independently, based on the data's relation to the current goal, whether it should be activated at this moment. The attention mechanism operates only at this layer — it only awards, and no longer inspects qualifications.

After ascension, the system architecture changes from:

```
All data → [single-scalar ranking] → top-K → inject into context
```

to:

```
All data → [Readiness vetting] → candidate set → [activation ranking] → top-K → inject into context
                 ↓ (failed)
            processor handling → re-review → candidate set / exclusion
```

### 3.3 Recursive Ascension: The Recursive Openness of Dimensional Elevation of Strategy

Dimensional Elevation of Strategy is not a one-time operation. It is naturally recursive — whenever the system faces a new compressed orthogonal dimension, the ascension operation itself can be applied again.

The first ascension recognized that "semantic classification" and "processing state" were compressed into the same classification label, and separated processing state into an independent criterion (HEEL). The second ascension recognized that "qualification judgment" and "activation scheduling" were compressed into the same composite score, and separated qualification and activation into sequential decision layers (Readiness–Activation Separation). The third ascension recognized that "configuration management" and "the reasoning process" were compressed into the same system boundary, and elevated configuration management into internal memory operable by the LLM (Configuration-as-Memory). The fourth ascension recognized that "identity continuity" and "session transience" were compressed into the same session container, and liberated identity from the session (Character decoupling).

Each ascension liberates the reasoning ability from a narrower decision space by one more layer. And higher-order cognitive behaviors — self-triggered deep refinement, cross-Character experience inheritance, proactive Shícè — are the natural products of these ascensions concatenated at runtime. So-called emergence is in fact the continuation of reasoning at a higher dimension.

### 3.4 The Universality of the Meta-Principle

The power of the Wall-Breaking Principle lies in this: it not only mathematically proves "without ascension, degeneration", but also provides an operational test criterion for "how to ascend", and a unified theoretical perspective for "the recursion after ascension". It applies to agent memory management, and equally to any system that inadvertently compresses multiple orthogonal decision dimensions — including but not limited to educational evaluation systems, organizational management structures, and resource scheduling algorithms.

Wall-hitting and wall-breaking constitute the two forms of updating. Parameter update revises beliefs within a given hypothesis space via Bayesian updating — this is the part covered by existing theory. Structural update, when parameter update reaches its end and the system repeatedly hits an unbreakable performance ceiling, triggers a qualitative change: abandon the old space, construct a new space, and restart parameter updating within the new space. The lived experience of wall-hitting — this felt sense of being stuck — is not subjective frustration but an objective signal that the old space is insufficient. Wall-hitting is the prelude to wall-breaking; wall-breaking is the dimensional expansion of updating. The two forms of updating alternate and advance, constituting recursive cognitive evolution.


## 4. The AIOS Cognitive Kernel: Four Strategic Elevations of Strategy Dimension

If existing AIOS frameworks manage "machine resources" (CPU, memory, storage), then the AIOS Cognitive Kernel manages "cognitive resources" (memory, attention, configuration, identity). These resources have unique properties: they are not statically allocatable units but are dynamic, mutually coupled, and require the system to possess self-awareness of them. The following four ascensions unfold these four compressed cognitive dimensions one by one, establishing independent decision layers.

The following four modules constitute the v1.0 core components of the AIOS Cognitive Kernel. The interface of each module is locked, but its internal implementation can be replaced or optimized by subsequent research. The marking of module boundaries follows this paper's core extension principle: any new module must identify a previously compressed orthogonal dimension and establish an independent sequential decision layer for it.

Each of these four ascension operations accomplishes two things at once: on the cognitive dimension, it separates a pair of orthogonal dimensions (semantics and state, qualification and activation, configuration and reasoning, identity and session); on the engineering implementation, it replaces a pre-compiled static structure with dynamic data operable by the LLM at runtime (semantic labels → Readiness criterion; composite score → sequential two-layer decision; external configuration file → Experience subclass; session container → three-layer Character–Fenshen–Yinao decoupling). The former answers "what was separated"; the latter answers "by what means it was separated" — the two sides describe the same ascension operation.

### 4.1 The First Ascension: HEEL Cognitive Memory — Separation of Semantics and Processing State

**The compressed dimension**: traditional agent memory systems share an implicit premise — the semantic type of data ("what kind of data is this") determines its management policy ("how to store it, retrieve it, and evict it"). A piece of dialogue, because it is "dialogue history", is forced to live and die with the session; a persona setting, because it is a "system prompt", stays forever at the top of the context even as it crowds out precious cognitive space.

Semantic type and processing state are compressed into the same classification label — the system cannot distinguish "a dialogue containing long-term preference information" from "a piece of pure transactional noise". Both are "dialogue" and therefore receive the same management policy.

**The ascension operation**: the HEEL framework replaces semantic classification with the **Readiness criterion** $\mathcal{I}(d)$ as the fundamental criterion of memory stratification. Readiness asks only "what processing state is the data in" — can it be placed into the context window directly without additional processing — and does not ask "what does the data mean".

From this criterion a four-layer memory structure necessarily follows:

| Layer | Criterion | Definition | OS counterpart |
| :------------- | :------------------------ | :--------------------------------- | :----------------- |
| **History**    | $\mathcal{I}=0$, raw conversational material | Raw dialogue logs, append-only | Disk storage |
| **Experience** | $\mathcal{I}=1$, ready-to-use finished units | Data units that are ready and directly injectable into context | Page cache / shared libraries |
| **External**   | $\mathcal{I}=0$, raw external material | External data sources requiring retrieval/parsing | Peripheral I/O |
| **Logic**      | — | Pure policy layer managing the read/write/evolution/compression of the first three layers | Memory management unit (MMU) |

**Key insight**: classification is determined not by what data "means" but by "what state it is in" — can it be consumed directly, or does it need processing? Two pieces of data with identical semantic content — the raw dialogue transcript of "the user is allergic to a certain drug" and the extracted structured fact — belong to different layers in HEEL: the former in History (raw material), the latter in Experience (finished product), because their processing states differ, irrespective of the semantics they share.

This is the **first ascension** of the policy dimension: liberating the processing state of data from compression by semantic labels, and establishing for it a judgment dimension independent of semantics. Data is no longer managed according to "what it is".

**Interface status: v1.0 core module, interface locked**
- Defined: the classification criterion of the four-layer memory, inter-layer flow rules
- To be extended: optimal algorithms for the intra-layer eviction policy (currently a pluggable policy; the interface is open)

### 4.2 The Second Ascension: Readiness–Activation Separation — Separation of Qualification and Activation

**The compressed dimension**: HEEL solves the classification problem — data now knows what processing state it is in. But it does not answer the question that follows immediately: **among the processed data, which should be activated into the context window right now?**

Readiness $\mathcal{I}(d)$ governs qualification — whether data can enter the context. But qualification is not activation — whether a ready datum ($\mathcal{I}=1$) should enter the context right now depends on its relation to the current goal. These two dimensions are compressed, in HEEL's composite scoring formula, into the same set of weights — semantic similarity participates both in judging "the data is relevant to the task" (activation propensity) and implicitly as a Readiness signal that "the data is well formatted".

**The ascension operation**: separate Readiness judgment from activation judgment into two independent sequential decision layers:

- **Readiness vetting layer**: independent of the current goal, judges solely on the data's size, format, and semantic self-containedness whether it meets the basic conditions for injection. This layer is pure data governance — it does not ask "is this datum important?", only "is this datum usable?".
- **Activation management layer** (Concentration Domain criterion): within the candidate set that has passed qualification vetting, judges solely on the data's relation to the current execution goal whether it should be activated. The attention mechanism operates only at this layer — it only awards, and no longer inspects qualifications.

Two layers in sequence, mutually non-determining. The awarder is no longer simultaneously the inspector.

**Orthogonality proof**: crossing the two criteria yields four non-empty logical quadrants:

|                               | $\text{Relevant} > \theta$ | $\text{Relevant} \leq \theta$ |
| :---------------------------- | :------------------------- | :---------------------------- |
| **$\mathcal{I}=1$ (ready)** | Should be activated; enters context | Not activated; remains on standby in Experience |
| **$\mathcal{I}=0$ (unprocessed)** | Triggers the processing pipeline | Not activated; remains in its original layer |

All four quadrants are non-empty — ready but irrelevant (summaries of completed subtasks), highly relevant but unprocessed (raw external documents), ready and relevant (the experience currently needed), unprocessed and irrelevant (old task history) — proving that the two criteria are logically fully independent.

**Application of the Wall-Breaking Principle**: the second ascension directly applies the principle's core corollary — the Readiness and activation dimensions separated here correspond to the Quality (Q) and Relevance (R) dimensions defined in [2]. The principle rigorously proves that when qualification and activation are compressed into the same scalar (dimensional conflation), the system's discrimination accuracy has an insurmountable structural ceiling. Readiness–Activation Separation is the only antidote revealed by the principle — re-expand the compressed dimensions, build an independent decision layer for each, and break through the structural ceiling.

**Interface status: v1.0 core module, interface locked**
- Defined: the two-layer sequential decision architecture, the Concentration Domain criterion
- To be extended: optimal allocation strategies for the attention budget (listed as an open problem in Section 7)

### 4.3 The Third Ascension: Configuration-as-Memory — Separation of Configuration Management and the Reasoning Process

**The compressed dimension**: the first two ascensions liberated memory management from semantics and sessions, and separated Readiness qualification vetting from activation scheduling. But the agent's operating parameters — model selection, temperature, role descriptions — are still compressed in external configuration files, completely isolated from the agent's reasoning process.

Configuration is by default a "static preset at the system layer" — loaded at startup, immutable at runtime, and inaccessible to the agent's reasoning. The agent can think about "what did the user ask", can retrieve "what do I remember", but cannot think "which model should I use to answer this question" — this decision is compressed outside the LLM's reach.

**The ascension operation**: migrate configuration parameters from external configuration files into the agent's memory system — specifically, the Experience layer — making them memory data directly readable and writable by the LLM.

The theoretical legitimacy of this migration lies in the following: model names, temperature values, role descriptions, capability labels — all satisfy the Readiness condition. They are structured text, tiny in size, semantically self-contained. To the LLM they have the same informational properties as any line of text in the dialogue history: a piece of text that can be read, understood, judged, and modified.

After migration, the agent gains a **self-referential plane of operation** — its reasoning can be directed not only at external tasks but also at its own mode of operation:

- **Intelligent routing**: the agent reads the model capability mapping table from the memory layer and autonomously selects the optimal model in light of task complexity.
- **Multimodal orchestration**: the agent parses the modality requirements implicit in user instructions, queries the capability mapping table for the corresponding models and tools, and plans invocation sequences in real time.
- **Self-organizing role ecology**: after analyzing task requirements, the agent autonomously proposes role design schemes and invokes tools to create multiple roles with independent memory partitions, independent models, and temperatures.

**Recursive display of ascension**: this is the third ascension. The first separated processing state from semantics, the second separated activation qualification from processing state, and the third separated configuration management from external presets. Each ascension lets the agent's reasoning touch a dimension previously compressed outside the reach of reasoning. And self-triggered deep refinement — the agent autonomously assembling a multi-role exploration team before answering, reviewing in parallel, and refining iteratively — is precisely the full unfolding of these three ascensions concatenated at runtime.

**Interface status: v1.0 core module, interface locked**
- Defined: Experience storage of configuration, LLM-readable/writable tool interfaces
- To be extended: automated detection and correction of configuration drift (listed as an open problem in Section 8)

### 4.4 The Fourth Ascension: Character–Fenshen–Yinao Decoupling — Separation of Identity Continuity and Session Transience

**The compressed dimension**: in traditional session-centric architectures, the agent's identity, memory, and reasoning engine are compressed into the same session container. When the session ends, the identity instance is destroyed with it; a new session means a new identity, unless the user explicitly carries over the persona settings. An agent's "who am I" is bound to the temporary boundary of "this conversation".

**The ascension operation**: decouple the identity architecture into three orthogonal semantic layers:

- **Character**: a persistent persona identifier bound to a lifelong memory repository $\mathcal{M}$. A Character's memories, preferences, and behavioral constraints do not change with the opening or closing of any session. The Character is the persistent answer to "who am I".
- **Fenshen**: a Character's temporary view under a specific task — one concrete subset selection of the Character's memory repository $\mathcal{M}$. A Fenshen maintains independent runtime state and interaction history, but the persona continuity and long-term memory of its parent Character are shared and persistent.
- **Yinao**: the Character's brain — the module responsible for managing model invocation, sensing reasoning state, and converting context into understanding output; hot-swappably replaceable. Replacing GPT-4 with Claude changes only the reasoning style, not the Character's memories, preferences, or persona continuity. The Yinao is "what I am thinking with right now", the Character is "who I am", and the Fenshen is "what I am doing right now".

Three orthogonal layers — changing one requires no change to the others.

**Completion of the ascension**: the fourth ascension liberates identity continuity from session transience. In traditional architectures these two dimensions are compressed into the session container — the session is simultaneously the identity boundary, the memory boundary, and the engine binding. Character decoupling elevates identity into a persistent cross-session entity; the session is demoted to a parameterization of a Fenshen — a temporary Concentration Domain view.

**Interface status: v1.0 core module, interface locked**
- Defined: the three-layer orthogonal identity architecture, the Fenshen temporary-view mechanism
- To be extended: cross-Yinao persona-continuity metrics (listed as an open problem in Section 7)

## 5. The Unification Argument: Why This Is "One" Kernel, Not "Four" Modules

### 5.1 The Shared Paradigm-Shift Logic

The four sub-frameworks share the same paradigm-shift operation — replacing "pre-compiled static structures" with "dynamic data operable by the LLM at runtime":

| Sub-framework | Old paradigm (dimensional conflation) | New paradigm (Dimensional Elevation of Strategy) | Shift logic |
| :---------------- | :------------------------- | :-------------------------------- | :------------------------- |
| HEEL              | Semantic classification determines management policy | Readiness determines management policy | Decouple data management from semantics |
| Readiness–Activation Separation | A composite score proxies both qualification and activation | Qualification review and activation scheduling are independent and sequential | Decouple attention-awarding from qualification inspection |
| Configuration-as-Memory | Configuration files are external presets | Configuration is internal memory | Unify configuration management with the reasoning process |
| Character decoupling | Character = prompt + session | Character = persistent persona + independent memory + independent configuration | Decouple identity from session |

**Unified formula**:

> **Replace "pre-compiled" static structures → with dynamic data operable by the LLM at runtime**

This unified paradigm-shift logic is, at a more fundamental level, the concrete engineering realization of the Wall-Breaking Principle. Replacing "pre-compiled static structures" with "dynamic data operable by the LLM at runtime" is essentially the Dimensional Elevation of Strategy operation — re-expanding the dimensions originally compressed in static structures, making them dynamic dimensions that the LLM can independently perceive and manipulate.

This unified formula corresponds precisely to the four ascension operations described in Chapter 4:

| Ascension | What was separated (cognitive dimension) | By what means it was separated (engineering implementation) |
| :---------------------- | :--------------------------- | :-------------------------------------------- |
| First (HEEL)          | Processing state from semantic labels | Semantic classification label → Readiness criterion (computed dynamically at runtime) |
| Second (Readiness–Activation) | Activation qualification from processing state | Composite scoring function → sequential two-layer decision (two independent layers) |
| Third (Configuration-as-Memory) | Configuration management from the reasoning process | External YAML/JSON → Experience subclass in the memory layer |
| Fourth (Character decoupling) | Identity continuity from session transience | Session container → three-layer persistent architecture of Character–Fenshen–Yinao |

"What was separated" is the cognitive kernel of ascension — looking "downward" to identify the compressed orthogonal dimension. "By what means it was separated" is the engineering manifestation — looking "upward" to replace pre-compiled static structures with dynamic data operable by the LLM at runtime. The two describe the same operation: **looking downward one sees the re-expansion of dimensions; looking upward one sees a paradigm leap in the mode of implementation.**

### 5.2 Interdependence Proof

The four sub-frameworks are not optional modules but necessary components of a complete AIOS Cognitive Kernel. Their interdependence is argued below:

**If only HEEL (without Readiness–Activation Separation)**: the agent has stratified memory but no activation management. Ready data in the Experience layer would be indiscriminately injected at every context construction — qualification management complete, activation management absent. In long-horizon tasks, the context would be eroded by "ready but irrelevant" data. This is an incomplete state of ascension — the first ascension done, the second not yet executed.

**If only Readiness–Activation Separation (without HEEL)**: the agent has the qualification/activation separation but no persistent stratified memory. All data still lives in session containers; cross-session memory cannot precipitate into the Experience layer. Qualification judgment lacks a storage foundation — at every new session the system must re-vet the Readiness of all data.

**If only Configuration-as-Memory (without HEEL)**: configuration is modifiable but not persistent. Across sessions, all agent-modified configuration is lost. The lifecycle of configuration memory would equal that of a session, which itself violates the principle of "configuration as memory" — memory should be persistent.

**If only Configuration-as-Memory (without Character decoupling)**: if one Character modifies the configuration, all Characters are affected — because configuration is not isolated per Character. Character A raises the temperature for creative writing, and Character B's scientific reasoning becomes correspondingly diffuse.

**All four present**: the agent possesses a complete cognitive operating-system kernel. Four-layer memory provides persistent stratified storage (HEEL); Readiness vetting and activation scheduling are independent and sequential (Concentration Domain); configuration is autonomously operable by the LLM in the memory layer and persistent across sessions (Configuration-as-Memory); identity is independent of sessions, and configuration modifications are isolated per Character (Character decoupling). Memory persistent, qualification clear, activation precise, configuration self-adaptive, identity continuous — cognitive resource management across all five dimensions operates independently yet cooperates as one.

**This proves the four are not "optional modules" but necessary components of a complete AIOS Cognitive Kernel.** With any one missing, the system develops a specific structural vulnerability. With all in place, the system reaches the complete state of a cognitive operating system.

Existing AIOS frameworks generally focus on task scheduling and resource orchestration (corresponding to part of the functionality of this paper's second ascension), but lack cognitive memory management (first ascension), self-referential configuration (third ascension), and identity persistence (fourth ascension). With any one missing, the system develops a specific structural vulnerability. The four ascensions of this paper constitute the necessary and sufficient components of a complete AIOS Cognitive Kernel.

Leading model vendors have made great progress in model capability, but generally face two common bottlenecks: the models' own "forgetfulness" (no persistent memory) and "lack of self-knowledge" (no meta-cognitive boundaries). ## 7. Interfaces to Be Extended and Future Ascension Directions

The following sections are explicitly marked "to be extended" in the current interface specification. They are not omissions of this paper but interfaces reserved for subsequent research — each direction corresponds to a cognitive dimension that has been identified but not yet fully unfolded. Filling these directions will constitute the iterative path of the AIOS Cognitive Kernel from v1.0 toward higher versions.

**7.1 Optimal allocation strategies for the attention budget**: after ascension, Readiness vetting and activation management are separated. But within the activation management layer, when multiple ready candidate blocks have similar activation scores and the context window is limited, what is the optimal allocation strategy? This corresponds to working-set selection optimization in operating systems.

**7.2 The trust axiom**: the four axioms of the AIOS Cognitive Kernel (existence, Readiness, metabolic necessity, Concentration Domain) cover all dimensions of cognitive resource management. But the essence of the cognitive kernel is **delegation** — the user delegates the management of cognitive resources to the system. Should a fifth axiom — a trust axiom — be established to formalize this delegation relation?

**7.3 ARIES (Agent Resource Information Exchange protocol)**: how is memory shared between different agents? How do configurations interoperate? When multiple AIOS instances form an agent society, the cross-instance exchange of cognitive resources needs a standardized protocol.

**7.4 The security model of the cognitive kernel**: where is the permission boundary for the LLM operating on configuration? Which operations are "kernel mode" and which are "user mode"? The permission model of a self-referential system needs a formal definition.

**7.5 Cross-Yinao experiential consistency metrics**: when the underlying model (Yinao) is replaced, how does the AIOS Cognitive Kernel guarantee that upper-layer behavior does not drift? How should a metric of a Character's "persona continuity" under cross-model conditions be defined?

**7.6 Detection and correction of configuration drift**: once the agent gains self-referential configuration capability, it may slowly deviate from the user's true preferences. How to distinguish "beneficial adaptation" from "harmful drift"? How to prevent configuration from running out of control without stripping the agent of its adaptive capability?

**7.7 The cognitive security model of AIOS**: once the system has self-referential configuration capability, how to distinguish "beneficial adaptation" from "harmful drift"? How should the permission model of a cognitive operating system be defined — which operations are "kernel mode" and which are "user mode"?

**7.8 Cross-Yinao cognitive continuity**: when the underlying model is replaced, how do upper-layer cognitive behaviors remain stable? How should a Character's "persona continuity" metric under cross-model conditions be defined?

**7.9 Consensus protocols for distributed cognitive kernels**: when multiple AIOS instances form a cognitive society, how do they exchange memory, configuration, and attention resources? The standardization needs of the ARIES protocol.


## 8. Discussion and Conclusion

### 8.0 A Higher-Order Open Problem of the Kernel: The Semantic-Understanding Dependency

This kernel has a shared dependency that must be candidly acknowledged, spanning all four ascension operations. The agent's semantic understanding of configuration items and memory items is the implicit premise of the entire kernel. The first ascension assumes the agent can judge whether a datum is "semantically self-contained"; the second assumes it can judge whether a datum "serves the current goal"; the third assumes it understands the functional meaning of configuration parameters such as model names and temperature values; the fourth assumes it understands the cognitive functions of different role perspectives.

If this premise is broken in any one ascension, the products of that ascension are built on systematic bias. And because the post-ascension architecture is sequentially transitive — the first ascension's candidate set enters the second ascension, and the second's activation results affect the third's configuration decisions — a single semantic-understanding bias may be amplified level by level by subsequent ascension operations.

It is worth noting that the old paradigm, lacking self-referential configuration capability, cannot even encounter the problem of "configuration drift". What this paper addresses is a new problem at a higher dimension, not a continuation of an old one. This hidden risk is not automatically eliminated by the architecture's completeness; it requires a joint monitoring mechanism across ascension operations — this is the central challenge the entire kernel must face together.

### 8.1 Signs of a Paradigm Shift

In *The Structure of Scientific Revolutions* [13], Kuhn emphasizes that the mark of a new paradigm is not "solving old problems better" but "making old problems no longer asked, while making new problems visible".

When the AIOS Cognitive Kernel is accepted:
- "How to compress memory" → "how to do incremental updates to the Experience layer"
- "How to manually configure agents" → "how the agent understands your configuration preferences"
- "How to create multi-agent teams" → "how the agent designs teams by itself according to the task"
- "How to maintain persona across sessions" → "Character Experience persistence"
- "How to prevent context bloat" → "Concentration Domain management — separating Readiness vetting from activation scheduling"

**The language of the questions themselves has changed — that is a paradigm shift.**

### 8.2 Conclusion

In the introduction we depicted the "parts market" predicament of current AIOS development. The AIOS Cognitive Kernel of this paper does not add a new stall to this market. It is a unified set of market rules — weights and measures, and traffic rules.

**Weights and measures**: the Wall-Breaking Principle. It quantitatively proves — an undecoupled system has its decision accuracy degenerate to the level of random guessing in the most extreme case. This is not "some part not being good enough" but "the weights and measures themselves being distorted" — when a system uses one ruler to measure multiple orthogonal dimensions, the measurement results are logically unreliable.

**Traffic rules**: Readiness vetting → activation scheduling, executed sequentially. Qualification judgment first, activation scheduling after. Ask first "can it?", then "should it?".

The AIOS Cognitive Kernel Interface Specification v1.0 released in this paper is a still-growing framework. Its four core modules have been validated by a reference implementation, its inter-module interfaces defined, its unified extension principle established. But the complete territory of the AIOS Cognitive Kernel is far from finished — the interfaces explicitly marked for extension in this paper, and the higher-order cognitive dimensions not yet identified, will constitute the iterative content of v1.1, v2.0, and versions beyond. This specification is not a wall, but a door.

HEEL Cognitive Memory, Readiness–Activation Separation, Configuration-as-Memory, Character decoupling — the four are not four new stalls, but the concrete instantiation of this set of weights and measures and traffic rules in four key dimensions of AIOS architecture. Four strategic dimensional ascensions, four re-expansions of compressed dimensions. Once all four are in place, the parts market disappears naturally — because all parts are assembled according to the same set of rules, and the interfaces between them are unified, predictable, and automatically cooperative.

Once this kernel is established, the agent is no longer "a chatbot that can call tools" but **a cognitive subject with the capacity for self-management**. It knows what it remembers, what state it is in, and with what resource configuration to face what task. It is not an assembled collection of parts, but a cognitive operating system with endogenous order.

**We have not added a new stall to the AIOS parts market. We have defined the cognitive kernel of AIOS — unifying the weights and measures and the traffic rules. This is the paradigm leap of AIOS from a "task operating system" to a "cognitive operating system".**
