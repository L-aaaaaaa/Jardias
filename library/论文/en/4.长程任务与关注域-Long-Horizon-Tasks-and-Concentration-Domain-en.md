# From Memory Eligibility to Attention Activation: The Concentration Criterion as the Second Orthogonal Principle for Agent Cognitive Resource Management

## Section 0. Conceptual Framework: The OS Memory-Management Mapping Table

Before developing the theory, we first anchor the top-level design philosophy of the entire paper.

> **Core insight**: Traditional agent designs ask, "How can we fit more information into this limited space?" The present approach asks instead—**why should we still tolerate seeing only this much space? We should build a virtual memory system for the agent.**

The agent's context window and a computer's physical memory are structurally isomorphic. Once this is recognized, a complete engineering mapping can be established between the two:

| Operating-system concept       | Agent cognitive resource management      | Corresponding mechanism in this work                         |
| :----------------------------- | :--------------------------------------- | :----------------------------------------------------------- |
| **Physical memory limit**      | Context window size                      | The hard constraint on a single inference—the physical boundary of the agent's cognitive resources |
| **Virtual memory**             | Logical memory space                     | The cognitive space built by HEEL's four-layer memory, far larger than the physical window |
| **Page table**                 | TaskState + concentration-domain mapping | Maintains the mapping of "which memory pages the current step needs" |
| **Page fault**                 | Triggering retrieval/compression         | When needed information is not in context, it is loaded from the External or History layer |
| **Working set**                | Current concentration domain $W_Q$       | The set of memory fragments actually activated in the context window |
| **Page replacement algorithm** | Concentration-domain recalibration       | Intelligent replacement based on semantic relevance, rather than naive FIFO |
| **Memory leak**                | Linear context bloat                     | Data that has exited the concentration domain keeps occupying cognitive space until overflow |
| **Process isolation**          | Goal anchoring                           | Prevents irrelevant thinking from contaminating the current context and derailing the task |

The fundamental significance of this table is not rhetorical analogy. It reveals a long-overlooked fact: **context management in agents is, in essence, the operating-system memory-management problem.** An operating system manages computational resources (CPU, memory, I/O); an agent manages cognitive resources—the context window is its physical memory, the memory store is its virtual memory space, and decisions about context construction are, in essence, the decisions of a page replacement algorithm: at this moment, which memory pages should remain in precious physical memory?

The central criterion of this paper—the Concentration Criterion—is precisely the cognitive-resource mapping of the OS concept of "working-set selection" onto agents. The question it answers is: **given the current execution goal, at this very moment, which pieces of ready memory data should be activated?**

## Abstract

**Background**: When executing long-horizon tasks, agents face a peculiar difficulty—goal drift and context bloat. Existing memory architectures (HEEL included) have solved the problem of "what processing state a piece of data is in," but have not systematically addressed the question of "among the data that is ready, which should be activated right now." These two dimensions are orthogonal, because the questions they answer are independent: one concerns an intrinsic property of the data; the other concerns a relational property between the data and the goal.

**Core claim**: This paper proposes the **Concentration Criterion** as a second principle of cognitive resource management, orthogonal to the Readiness Criterion. Readiness $\mathcal{I}(d)$ judges the processing state of data (**readiness**: whether it *can* be injected); the Concentration Criterion $\text{Relevant}(d, Q)$ judges the relation between the data and the current execution goal (**activation**: whether it *should* be injected right now). The division of labor is clear; the two are mutually independent yet complementary—together they constitute a complete theoretical foundation for agent cognitive resource management.

**Formalization**: We define the concentration domain as the subset of ready data in the memory store whose relevance exceeds a threshold, $W_Q = \{d \in \mathcal{M} \mid \text{Relevant}(d, Q) > \theta, \mathcal{I}(d)=1\}$, and establish Axiom 3 (the Concentration Necessity Axiom): at any moment, an agent's context window should hold only a subset of the current concentration domain. Data outside $W_Q$, whatever its processing state, should not occupy context space.

**Corollaries and engineering instantiation**: Three corollaries follow necessarily from Axiom 3—(1) context occupancy does not grow in proportion to the number of steps (because the size of $W_Q$ depends on the goal's relevance distribution rather than on the number of elapsed steps); (2) goal anchoring is a correctness guarantee for concentration-domain computation (without anchoring $Q$, $W_Q$ cannot be computed); (3) the state file is the engineering instantiation of the concentration domain (its "Goal" section anchors $Q$; its "Completed" section marks data that has exited $W_Q$; its "Key Findings" section pins critical pages). The state file further introduces **Confidence Tags** (🟢 High Confidence / 🟡 Speculative / 🔴 Uncertain)—the agent's ex-ante declaration of the certainty of its cognitive outputs, ensuring that autonomous execution always proceeds within knowable and controllable cognitive boundaries. Confidence Tags touch the core bottleneck of AI reliability: not "will the agent make mistakes," but "does the agent know where it might make mistakes."

**Validation**: A minimal sufficient experiment directly tests the core hypothesis—injecting only an explicit state file to force the anchoring of $Q$ and $W_Q$ significantly reduces the rate of goal drift. An inversion test demonstrates that the state file indeed functions as a concentration-domain anchoring mechanism.

**Positioning**: The Concentration Criterion completes the dimension left uncovered by the HEEL axiom system—the management of memory "activation." Readiness + Concentration + Metabolic Necessity correspond to three independent subsystems of OS memory management: the MMU (readiness check), the page replacement algorithm (activation selection), and disk cleanup (metabolic elimination). The peculiar difficulty of long-horizon task execution proves not how this theory can be applied, but why this theory must exist.

## 1. Introduction

### 1.1 Three Scenarios: The Failure of Context Management

**Scenario 1: Goal drift.** The user says: "Analyze the profit anomalies in this financial report and find the cause." At step 5 the agent begins analyzing overall financial health; by step 12 it has completely forgotten the original anchor of "profit anomalies" and has moved on to a full-scope analysis. After 50 conversational turns, the context window is filled with intermediate reasoning, while the original goal is diluted to near-untraceability.

**Scenario 2: Context erosion.** The agent handles a complex task requiring multi-step execution. Each step produces new intermediate results, which are compressed and stored in the Experience layer. These compressed artifacts are all semantically related to the task—they come, after all, from discussions of the same project—but most of them serve subtasks that are already finished. When the agent reaches step 20, 60% of the context window is occupied by "highly ready but irrelevant" data from the first 15 steps; only the remaining 40% actually serves the current step.

**Scenario 3: Attention dilution.** In a multi-role collaboration task, Role A finishes data collection and writes to the shared Experience. Role B starts the analysis phase, and its context builder retrieves "relevant memories" from the Experience layer—yet what gets activated includes not only the data Role A delivered (which is needed) but also Role A's intermediate reasoning and error-correction records from the collection process. These records are not merely useless to Role B; they are distracting. Worse, acting on some of the error-correction records, Role B repeats directions that Role A had already ruled out.

The shared pathology of the three scenarios: **the context window—the agent's scarcest cognitive resource—is allocated indiscriminately to all historical data, rather than precisely to the information that the current execution goal truly needs.** This is not a "window too small" problem. Even if the window were enlarged tenfold, without a correct activation strategy the system would still fill it up—no longer with 50 steps of garbage, but with 500 steps of garbage.

### 1.2 Core Insight: An Overlooked Dimension

Consider the current state-of-the-art memory architectures—HEEL, MemGPT, Letta. Collectively, they address the dimension of "what processing state the data is in." HEEL's Axiom 1 (the Readiness Criterion) strictly defines whether a piece of data has been processed to the point where it can be directly injected into the context window ($\mathcal{I}=1$ or $\mathcal{I}=0$), and accordingly sorts data into three layers: History (raw material), Experience (finished product), and External (external raw material).

This is a crucial contribution. But it leaves a natural follow-up question unanswered:

> A piece of data already satisfies $\mathcal{I}=1$—it has been processed and is qualified to enter the context window. But **should** it be activated right now?

**These two questions are logically independent.** "Can it" concerns the data's processing state (an intrinsic property); "should it" concerns the data's relation to the current execution goal (a relational property). An Experience entry such as "the user prefers concise answers" remains in the "ready" processing state whether the current task is writing a poem or checking the weather. Yet it should not be activated for the poem (it would inhibit creativity) and should be activated for the weather (in that scenario the user does prefer brevity). The processing state has not changed; the activation state changes with the goal.

The Readiness Criterion manages the **readiness** of data. But **activation**—this is another dimension, one that requires an independent criterion orthogonal to Readiness.

### 1.3 The Proposed Approach: Establishing the Concentration Criterion as an Independent Criterion

This paper proposes the **Concentration Criterion**:

> At any moment, an agent's context window should hold only ready data directly relevant to the current execution goal $Q$. Data beyond this scope, whatever its processing state, should not occupy precious physical cognitive space.

This criterion is the precise mapping of the operating system's "working-set model" onto agent cognitive resource management. Just as an operating system, through its page replacement algorithm, keeps only the pages in the current process's working set in physical memory, an agent's context builder should confine activation to the concentration domain $W_Q$—the data genuinely necessary for accomplishing $Q$, not all the historical information that "looks relevant."

This paper does three things:

1. **Establishing the criterion**: elevating the "concentration domain" from a derived function of HEEL to an independent criterion orthogonal to Readiness
2. **Theoretical development**: establishing Axiom 3 and deriving three engineering corollaries for long-horizon task execution
3. **Experimental validation**: directly validating the effectiveness of concentration-domain management through a minimal sufficient experiment (forced anchoring via the state file); ruling out alternative explanations through an inversion test

### 1.4 Contributions

1. **A new criterion**: establishing the Concentration Criterion and proving its orthogonality to the Readiness Criterion (all four logical quadrants are non-empty), thereby completing the theoretical closed loop of agent cognitive resource management
2. **Axiomatic extension**: adding Axiom 3 on top of HEEL's Axioms 0–2; the three axioms correspond to three independent subsystems of OS memory management
3. **A system of corollaries**: deriving from Axiom 3 three engineering corollaries—the non-growth of context occupancy, the necessity of goal anchoring, and the functional positioning of the state file
4. **Engineering concepts**: the state-file template and Confidence Tags—the agent's ex-ante declaration of its cognitive boundaries, touching the core bottleneck of human–AI trust
5. **Experimental validation**: a minimal sufficient experiment showing that forced concentration-domain anchoring alone significantly reduces the rate of goal drift

*The Concentration Criterion is the second instantiation of the metrology of the agent-OS paradigm: it separates "readiness" from "activation" into two sequential decision layers. It is the second step of the traffic rules—an activation judgment made on the basis of the current goal within the candidate set. The awarder is no longer also the inspector.*

## 2. Background and Problem Analysis

### 2.1 The Coverage of the HEEL Axiom System and the Dimension It Leaves Open

The HEEL framework establishes three axioms:

- **Axiom 0 (Existence Axiom)**: the agent possesses a lifelong memory store $\mathcal{M}$ that persists across sessions
- **Axiom 1 (Readiness Axiom)**: data $d$ is stratified by processing state—$\mathcal{I}(d)=1$ is Experience (directly injectable); $\mathcal{I}(d)=0$ is assigned to History or External according to its source
- **Axiom 2 (Metabolic Necessity Axiom)**: the memory store must undergo a continuous closed loop of encoding → consolidation → decay → elimination

These three axioms cover the **existence**, **processing state**, and **lifecycle** of memory. But one dimension has not been systematically addressed: **among the data that has been processed, which should be activated into the context window right now?**

When the ContextBuilder retrieves candidate chunks from the Experience layer, it uses the composite scoring formula:

$$score = \alpha \cdot \text{Sim}(q, d) + \beta \cdot \text{Rec}(d) + \gamma \cdot \text{Imp}(d)$$

This formula works well in short-horizon dialogue, where the current query $q$ almost coincides with the current goal $Q$. In long-horizon tasks, however, $q$ and $Q$ diverge systematically—$q$ is "what is being asked at this moment" (the local semantics of the current substep), while $Q$ is "what should be done at this moment" (the core intent unchanged since the original goal). Using $q$ as the relevance reference, the composite score retrieves "memories semantically similar to the current local query" rather than "memories genuinely relevant to the current overall goal."

This is the gap between Axiom 1 and Axiom 3: **readiness management (processing state) is complete; activation management (the concentration domain) is still absent.**

### 2.2 Two Peculiar Difficulties of Long-Horizon Tasks

**Difficulty 1: Goal Drift.** In long-horizon execution, the agent does not act under a static $Q$. Each step's reasoning output modifies the system's perception of "what is currently being done," and the perceptual offsets accumulated over many steps can lead the agent to unknowingly substitute a drifted $q$ for the original $Q$ in context retrieval. Once retrieval is contaminated, the memory chunks it brings in further reinforce the drifted perception—a positive feedback loop. This is the mechanistic reason why agents, in long-horizon tasks, "chat along and forget what they were originally supposed to do."

**Difficulty 2: Context Erosion.** Even when $Q$ is correctly maintained, each step of execution still produces new History and Experience. These new data are indeed semantically related to $Q$ (they are, after all, products of the same task), but they actually serve completed subtasks rather than the current step. The composite scoring formula cannot precisely distinguish "relevant to the whole task" from "relevant to the step at hand"—these two kinds of "relevance" overlap heavily in vector space. The result: the more that gets done, the more severely the context is eroded by data that is "no longer needed but still looks relevant."

### 2.3 A Formal Distinction Between the Two Kinds of Drift

This paper distinguishes two types of drift. They share a common root (inaccurate concentration-domain computation) but differ in manifestation and mechanism:

- **Type 1 (goal drift)**: the reference $Q$ itself is replaced by $Q'$—the system is doing the wrong thing. Corresponding problem: the anchoring of $Q$ in the computation of $\text{Relevant}(d, Q)$ has failed.
- **Type 2 (attention drift)**: $Q$ is unchanged, but the boundary of $W_Q$ is eroded—the system is doing the right thing, but its attention is scattered by irrelevant information. Corresponding problem: the threshold $\theta$ or the scoring function of $\text{Relevant}$ fails to effectively distinguish "relevant to $Q$" from "relevant to completed subtasks of $Q$."

In current practice, goal anchoring addresses Type 1 (but lacks a theoretical foundation), and sliding windows address Type 2 (but perform replacement on the basis of time rather than semantics). The unified treatment architecture of this paper is developed in Section 5.

## 3. The Concentration Criterion: Definition and Formalization

### 3.1 Axiom 3 (the Concentration Necessity Axiom)

**Statement**:

> At any moment, an agent's context window should hold only a subset of the current concentration domain $W_Q$. Data outside $W_Q$, whatever its $\mathcal{I}$ value, should not occupy context space.

**Why this is an "axiom" rather than an "engineering strategy"**: the context window is the agent's scarcest cognitive resource—it directly determines how the model's attention is allocated at inference time. Forcing data that does not serve the current goal $Q$ into the context window not only wastes scarce physical space; more fatally, it dilutes the model's attention on genuinely relevant information. This cannot be regarded as "an efficiency problem that can still be optimized"—it allocates cognitive resources incorrectly at the level of principle, just as an operating system cannot allow the data pages of process A to forcibly occupy the physical frames of process B.

**This axiom does not modify Axioms 1–2; it supplements a dimension they do not cover.** What it addresses is not "what state the data is in," but "whether the data should be in an activated state right now." The relation between readiness (Readiness) and activation (concentration domain) receives its full formal argument in Section 3.3.

### 3.2 Formal Definition

**Definition (Concentration Domain $W_Q$)**: Given the current execution goal $Q$ and the lifelong memory store $\mathcal{M}$, the concentration domain is the subset of $\mathcal{M}$ satisfying both the Readiness condition and the relevance condition:

$$W_Q = \{d \in \mathcal{M} \mid \text{Relevant}(d, Q) > \theta_{\text{rel}}, \quad \mathcal{I}(d) = 1\}$$

where:

- $\mathcal{M}$ is the lifelong memory store (Axiom 0)
- $\text{Relevant}(d, Q)$ is the relevance scoring function of data $d$ with respect to the current execution goal $Q$, taking values in $[0,1]$
- $\theta_{\text{rel}}$ is the injection threshold (data whose relevance score falls below it does not enter $W_Q$)
- $\mathcal{I}(d) = 1$ ensures that all candidate data have passed the readiness review (satisfying Readiness)

**Key design decision**: the reference of $\text{Relevant}(d, Q)$ is $Q$ (the complete semantics of the current execution goal), not $q$ (the local query text of the current step). This difference of reference is exactly what distinguishes the Concentration Criterion in essence from the composite scoring formula—the composite score uses $\text{Sim}(q, d)$, whose reference $q$ itself drifts during execution.

### 3.3 Proof of Orthogonality Between the Two Criteria

To prove that Readiness $\mathcal{I}(d)$ and the Concentration Criterion $\text{Relevant}(d, Q)$ are two orthogonal dimensions, we examine the four logical quadrants produced by crossing them:

|                                   | $\text{Relevant} > \theta$                                   | $\text{Relevant} \leq \theta$                                |
| :-------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **$\mathcal{I}=1$ (ready)**       | **Quadrant 1**: ready and relevant now<br>→ should be activated; enters $W_Q$ | **Quadrant 2**: ready but irrelevant now<br>→ not activated; kept on standby in the Experience layer |
| **$\mathcal{I}=0$ (unprocessed)** | **Quadrant 3**: highly relevant but not yet processed<br>→ not activated; the processing pipeline is triggered instead | **Quadrant 4**: unprocessed and irrelevant now<br>→ not activated; remains in its original layer |

**All four quadrants are non-empty**:

- Quadrant 1: the user preference "concise answers," with the current task being a weather query—should be activated
- Quadrant 2: the summary of a completed subtask—processed, but not serving $Q$ at this moment
- Quadrant 3: a chapter of an external PDF highly relevant to $Q$—first retrieved/summarized → processed into Experience → enters Quadrant 1
- Quadrant 4: old task history unrelated to $Q$—stays in its original layer and takes no part in any processing

Since all four quadrants are non-empty, the two criteria are logically fully independent: neither determines the other's value. Hence the two criteria are orthogonal.

**The division of labor between the two criteria**:

| Dimension               | Readiness $\mathcal{I}(d)$                               | Concentration Criterion $\text{Relevant}(d, Q)$              |
| :---------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **Question asked**      | Can this datum be injected directly?                         | Should this datum be activated right now?                    |
| **Basis of judgment**   | The data's processing state (size, format, self-containedness) | The data's relation to the goal (semantic relevance, temporal relevance) |
| **Dependent variables** | $d$ itself (independent of $Q$)                              | $d$ and $Q$ (independent of processing state)                |
| **Frequency of change** | Changes only when the data is reprocessed                    | Changes continuously as $Q$ advances                         |
| **Object managed**      | The data's **readiness** state                               | The data's **activation** state                              |
| **OS counterpart**      | MMU (readiness check)                                        | Page replacement algorithm (activation selection)            |

The distinction between the corresponding mechanisms is the strongest evidence for orthogonality: an operating system has two fully independent subsystems, one governing "whether access is possible," the other governing "whether something should remain in physical memory." Agent cognitive resource management likewise requires these two independent subsystems.

### 3.4 The Dynamism of the Concentration Domain

The essential characteristic of $W_Q$ is dynamism—it changes continuously as $Q$ advances:

$$W_Q^{(t)} \neq W_Q^{(t+1)} \text{ if and only if } Q \text{ undergoes semantic advancement between steps}$$

When $Q$ advances from one stage to the next (e.g., from "collect data" to "analyze data"), data that served the previous stage but is no longer needed in the next may exit $W_Q$, while key results delivered by the earlier stage (which are inputs to the later one) remain in $W_Q$ as pinned pages.

This goal-advancement-driven evolution of $W_Q$ is the qualitative difference between the concentration domain and a naive sliding window—a sliding window performs replacement on the basis of time; the concentration domain performs replacement on the basis of goal semantics.

## 4. Corollaries: From the Concentration Criterion to a Long-Horizon Task Architecture

### 4.1 Corollary 1: The Non-Growth of Context Occupancy

**Statement**: In long-horizon tasks, context occupancy does not grow in proportion to the number of steps. The occupancy of the context window depends on the volume of data inside $W_Q$, and the size of $W_Q$ depends on the relevance distribution of $Q$, not on the number of elapsed steps.

**Derivation**:

1. The context window loads only a subset of $W_Q$ (Axiom 3)
2. The size of $W_Q$ is determined by the amount of $\mathcal{I}=1$ data satisfying $\text{Relevant}(d, Q) > \theta$
3. Data produced by a new step enters $W_Q$ only if its relevance to the current $Q$ exceeds the threshold
4. When $Q$ advances, subtask data serving the old $Q$ may exit $W_Q$ as its relevance decays
5. Therefore the size of $W_Q$ depends on the semantic distribution of $Q$ (a property that varies within a bounded range), not on the number of elapsed steps (a monotonically increasing property)

**What this means**: "permanent context headroom"—the principle that appears in the long-horizon task proposal as an initial design axiom—here acquires the force of a theorem. It is no longer an independent engineering goal requiring its own defense, but a logical corollary of Axiom 3: as long as $W_Q$ is correctly maintained, growth in the number of steps does not necessarily lead to context bloat.

### 4.2 Corollary 2: Goal Anchoring Is a Correctness Guarantee for Concentration-Domain Computation

**Statement**: The goal-anchoring mechanism—whereby the system forcibly injects $Q$ as an invariant reference at context-construction time—is not an add-on engineering optimization, but a necessary condition for computing $\text{Relevant}(d, Q)$.

**Derivation**:

1. Concentration-domain computation depends on the reference $Q$ (by definition)
2. In long-horizon execution, the agent may retrieve memory using a $q$ contaminated by the current step's semantics in place of $Q$ (analysis in Section 2.2)
3. If this substitution occurs, $W_Q$ degenerates into "the set of data relevant to the current locally similar query," decoupled from the original $Q$
4. The sole function of the goal-anchoring mechanism is to lock $Q$ as the stable reference for relevance computation
5. Therefore, to compute $W_Q$ correctly, $Q$ must be anchored

**What this means**: goal anchoring—the Supervisor's independent evaluation, the fixed "Goal" section injected in the state file, the re-injection of the original goal at context-construction time—is introduced not because "doing so is better," but because "without it, the Concentration Criterion has no correct basis of computation." It is not an engineering optimization, but an engineering consequence necessitated by the criterion.

### 4.3 Corollary 3: The State File = The Engineering Instantiation of the Concentration Domain

**Statement**: The MVP state-file scheme of the blueprint draft should not be understood as "a simplification made because the ideal architecture is not yet finished," but as the direct engineering realization of the Concentration Criterion. Every section of the state file corresponds to one link of $W_Q$ management.

The full structure of the state file and its correspondences are as follows:

```markdown
# Task State

## Goal
[The user's original task description]
→ Corresponds to: $Q$—the anchored reference for concentration-domain computation

## Completed
- ✅ 2026-01-15 10:30: Successfully scraped 200 article titles
- ✅ 2026-01-15 10:35: Completed text cleaning of the titles
→ Corresponds to: the exited portion of $W_Q^{(t)}$—marking data that no longer needs activation

## Current Stage
Performing keyword extraction on the titles (120/200 completed)
→ Corresponds to: the current core of $W_Q^{(t)}$—the index of currently activated data

## Key Findings / Constraints
- Titles contain many emojis; cleaning rules must cover Unicode emoji ranges
- The target website has anti-scraping defenses; the request interval must not be under 2 seconds
→ Corresponds to: pinned pages in $W_Q^{(t)}$—critical memory pages that must not be mistakenly evicted

## Confidence Tags
- 🟢 High Confidence: text cleaning complete; 200 titles successfully scraped
- 🟡 Speculative: estimated 98% success rate for keyword extraction, to be verified
- 🔴 Uncertain: word-cloud generation depends on the quality of the keyword results; priority review needed
→ Corresponds to: the confidence level of each element in $W_Q^{(t)}$—informing the system and the user of the boundary reliability of the current concentration domain
```

**Formal definition of Confidence Tags**:

> **Confidence Tags** are the agent's ex-ante declaration of the certainty of its cognitive outputs. The system automatically labels every item in $W_Q$ with one of three levels:
>
> - 🟢 **High Confidence**: the tool returned success with verifiable results, or every step of the reasoning chain has a clear external basis
> - 🟡 **Speculative**: based on pattern inference or historical experience, but not externally confirmed within the current task
> - 🔴 **Uncertain**: relies on incomplete information, ambiguous reasoning, or equivocal interpretation, and carries a relatively high risk of error
>
> When the proportion of 🔴-tagged items in $W_Q$ exceeds a preset threshold $\theta_{\text{conf}}$, the system automatically suspends autonomous execution and requests user review.

Confidence Tags are not merely a UI design. Theoretically, they correspond to the boundary reliability of $W_Q$—the system's metacognition of its own cognitive boundaries. Their core value is not "letting the user see progress," but ensuring that autonomous execution always proceeds within knowable and controllable cognitive boundaries. At a time when the reliability of AI systems is widely questioned, an agent that can proactively and honestly declare "this part is my speculation; I suggest you review it" touches the core bottleneck of human–AI trust.

**What this means**: the state file is not an engineering compromise, but the simultaneous instantiation—in a user-readable, user-editable form—of the three core elements of the Concentration Criterion: anchoring of $Q$, boundary management of $W_Q$, and declaration of cognitive boundaries. Every operation the user performs by editing the state file is, in essence, a direct manipulation of $W_Q$: modifying the goal → correcting the reference $Q$; marking items completed → cleaning out data that has exited $W_Q$; writing down constraints → pinning critical pages; modifying confidence → calibrating the cognitive boundary.

## 5. Unified Treatment of the Two Kinds of Drift: Continuous Recalibration of the Concentration Domain

### 5.1 The Essence of Goal Drift: An Error of the $Q$ Reference

When the goal-anchoring mechanism is absent, the agent at each step uses the previous step's local query $q^{(t)}$ for semantic retrieval. Over multiple iterations, $q^{(t)}$ is continually updated into a new query based on the previous step's retrieval results and reasoning output. This recursive updating causes $q^{(t)}$ to drift gradually away from the original $Q$. Once the drift exceeds some threshold, the concentration domain $W_Q$ is replaced by a $W_{q}$ computed from the drifted $q$—the system is still computing a concentration domain, but the reference is no longer the original goal.

The treatment of goal drift is single and direct: **before every context construction, forcibly inject $Q$ (rather than the current $q$) as the stable reference for relevance computation.** This is the function of the state file's "Goal" section—it ensures that, no matter which step execution has reached, judgments of "relevant information" are always benchmarked against the original intent.

### 5.2 The Essence of Attention Drift: Erosion of the $W_Q$ Boundary

Even when $Q$ is correctly anchored, attention drift can still occur. When newly generated Experience keeps entering the fringe regions of $W_Q$—data that bears some semantic relevance to $Q$ (coming, after all, from the same task) but actually serves completed subtasks—the boundary of $W_Q$ exhibits a "creeping effect": more and more seemingly relevant data slowly floods in, while data serving completed subtasks is not marked as exited in time.

Attention drift requires a different treatment—**boundary cleanup of $W_Q$**. In the state file, every mark in the "Completed" section essentially tells the system: "this datum no longer needs to enter the current concentration domain." Explicit boundary recomputation at stage transitions further excludes the data of completed stages from $W_Q$.

### 5.3 The Unified Treatment Architecture

The two kinds of drift share the same root (inaccuracy of concentration-domain computation) and are handled separately by three mechanisms within a single architecture:

1. **$Q$ anchoring** (the state file's Goal section + goal re-injection before every context construction) → prevents goal drift
2. **$W_Q$ boundary cleanup** (marking in the Completed section + recomputation of $W_Q$ at stage transitions) → prevents attention drift
3. **Pinned pages** (Key Findings/Constraints + critical data marked 🟢 High Confidence) → prevents key information from being mistakenly evicted from $W_Q$

The three mechanisms are not a patchwork of independent features. They are the engineering instantiations of the Concentration Criterion under different failure modes (failure of $Q$ anchoring, erosion of the $W_Q$ boundary, eviction of critical pages). They share the same theoretical foundation—maintaining the correctness of $W_Q$ computation and the clarity of its boundary.

## 6. Relation to the HEEL Axiom System

### 6.1 Axiomatic Extension

This paper supplements HEEL's three axioms with Axiom 3 (the Concentration Necessity Axiom). The four axioms cover all dimensions of agent cognitive resource management:

| Axiom       | Object managed                     | Core criterion                             | Corresponding OS subsystem                            |
| :---------- | :--------------------------------- | :----------------------------------------- | :---------------------------------------------------- |
| Axiom 0     | The existence of memory            | Lifelong memory store $\mathcal{M}$        | Disk storage                                          |
| Axiom 1     | The processing state of memory     | Readiness $\mathcal{I}(d)$ (readiness) | MMU (readiness check)                                 |
| Axiom 2     | The lifecycle of memory            | Metabolic necessity                        | Disk cleanup/TRIM                                     |
| **Axiom 3** | **The activation state of memory** | **Concentration domain $W_Q$**             | **Page replacement algorithm (activation selection)** |

**The sequential relation of the two ascensions**: the orthogonal relation between the Readiness Criterion and the Concentration Criterion is the two sequential unfoldings of the Wall-Breaking Principle. The first ascension (HEEL, Axiom 1) separated processing state from semantics—establishing an independent criterion for whether data "can be used." The second ascension (this paper, Axiom 3) separates activation from readiness—establishing an independent criterion for whether ready data "should be used right now." The two ascensions are logically irreducible to each other: the first answers "what state is this datum in," while the second answers "among the data in the ready state, which should be activated right now." Their orthogonal relation—all four quadrants non-empty—is direct evidence of the recursive structure of strategic dimensional ascension: the candidate set produced by the first ascension is exactly the input space of the second.

The core theoretical value of Axiom 3 lies in **its explicit orthogonality to Axiom 1**. Axiom 1 ensures that only legitimate data can be accessed (readiness); Axiom 3 ensures that whatever is accessed is needed right now (activation). In an operating system, these two functions belong to two completely independent subsystems—the MMU and the page replacement algorithm; agent cognitive resource management likewise requires these two independent dimensions. Axiom 3 is not a modification of, or an explanatory note to, Axiom 1, but an independent new dimension logically orthogonal to it.

### 6.2 The Relation Between the Concentration Criterion and the Composite Score

The composite scoring formula $score = \alpha \cdot \text{Sim} + \beta \cdot \text{Rec} + \gamma \cdot \text{Imp}$ continues to exist as one of the methods for computing $W_Q$. The Concentration Criterion stipulates **what the computation should reference**—the relevance reference must be $Q$ rather than $q$, and what is computed is the service relation of $d$ to $Q$, not textual similarity to $q$. The composite score is responsible for **how to compute**—but with $\text{Sim}(q, d)$ replaced by $\text{Relevant}(d, Q)$. The criterion governs correctness (prescribing the correct reference); the formula governs efficiency (achieving the optimal ranking under that reference).

## 7. Experiment: A Minimal Sufficient Validation of Forced Concentration-Domain Anchoring

### 7.1 Core Hypotheses

**H1 (main hypothesis)**: Merely by explicitly injecting a state file to force the anchoring of $Q$ and $W_Q$, the rate of goal drift in long-horizon tasks can be significantly reduced, without relying on complex mechanisms such as a full Supervisor or pyramid planning.

**H2 (inversion hypothesis)**: If an incorrect anchor is deliberately written into the state file, the agent's behavior will be systematically controlled by the state file—demonstrating that the state file indeed functions as a concentration-domain anchoring mechanism rather than a mere record.

### 7.2 Experimental Design

**Test task set**: 50 custom multi-step tasks (averaging 15–20 steps), each containing 2–3 "intent-fork traps"—instructions in the task description that admit equivocal interpretation, such that an unanchored agent in its natural state has a high probability of choosing a deviant interpretive path. For example: "First organize last week's sales data, then analyze recent performance and see whether there are any anomalies"—"recent" may refer to last week (in the same vein as "last week"), this month (a more macro view), or this quarter (a strategic view); at step 5 the agent may leave the correct chain "last week + recent = last week" and instead take the initiative to analyze this month's data on its own.

**Comparison configurations**:

| Configuration                           | Context construction                                         | Goal anchoring                                               |
| :-------------------------------------- | :----------------------------------------------------------- | :----------------------------------------------------------- |
| **A: Vanilla ReAct**                    | Standard ReAct loop; only the current query + tool results; no anchoring of any kind | None                                                         |
| **B: Sliding window**                   | ReAct + injection of the most recent K turns of history      | Only residual traces of the original instruction in the history |
| **C: Full summarization**               | Configuration B + automatic summarization every 5 steps, with the summary injected | The summary may or may not retain the original goal          |
| **D: State-file anchoring (this work)** | Configuration C + injection of the state file as the leading block of the context at every construction | Forced anchoring: $Q$ is always at the very top of the context |

**Key point**: all four configurations use exactly the same HEEL memory backend and tool set. The only controlled variable is the presence or absence of state-file injection (and the forced anchoring of $Q$ that comes with it).

**Metrics**:

- **Decay curve of goal semantic similarity**: the semantic cosine similarity between the current execution goal and the original goal at each step (computed by an independent encoder)
- **Number of deviations and first-deviation step**: human annotation of whether a semantic deviation occurs at each step (3 annotators; Fleiss' Kappa reported)
- **Context token occupancy**: the actual number of tokens in the context at each step, measuring the relation between occupancy and step count
- **Task completion rate**: whether the original goal was achieved (three-valued: fully achieved / partially achieved / not achieved)

### 7.3 Inversion Test

**Design**: On a fixed task set, use Configuration D, but deliberately alter the "Goal" section of the state file to an incorrect goal deviating from the original intent (e.g., changing "analyze the profit anomalies" to "analyze the overall financial report"). Measure whether the agent acts according to the incorrect goal.

**Metric**: wrong-goal compliance rate—the proportion of tasks in which the agent executes according to the incorrect goal. A high compliance rate demonstrates that the state file indeed functions as a concentration-domain anchor; if the agent does not comply, this indicates that the agent itself has strong goal-retention ability.

## 8. Discussion

### 8.1 Why the Concentration Criterion Had Not Been Independently Proposed Before

In short-horizon dialogue, $Q$ is almost identical to $q$—whatever goal the user's utterance expresses is exactly the semantics of the current query. $W_Q$ and the retrieval results based on $\text{Sim}(q, d)$ overlap almost entirely; computing the concentration domain independently yields almost no additional information gain. The absence of the Concentration Criterion causes no visible loss.

Long-horizon tasks systematically expose the difference between $Q$ and $q$. The current query $q$ (which substep is being executed) and the current goal $Q$ (what the original intent is) can diverge measurably over multi-step reasoning. It is precisely this divergence that makes the necessity of the Concentration Criterion manifest—not that "the concentration domain was invented for long-horizon tasks," but that "long-horizon tasks make visible a dimension that has always existed but was masked by short-horizon dialogue."

### 8.2 Generalization of the Concentration Criterion

**Shícè (时策) scenarios**: when a Shícè task fires, $Q$ is not the user's current conversational goal but the goal of the scheduled task itself. Concentration-domain computation must be bound to that task's goal; otherwise, context from the user's conversation will contaminate memory activation when the scheduled task executes. This is the parallelism of two $W_Q$'s—the conversational $W_Q$ and the Shícè $W_Q$, each anchored to its own goal.

**Configuration-modification scenarios**: when the agent autonomously modifies configurations (e.g., adjusting temperature, selecting a model), the Experience on which the modification decision is based should come only from the $W_Q$ relevant to the current execution goal, not from all the historical data in the entire Experience layer. Otherwise, inappropriate adjustments may be made on the basis of irrelevant experience.

**Multi-role collaboration scenarios**: different roles should have different $W_Q$'s, even when they share the same Experience layer. Role A's internal reasoning process from data collection should not enter the $W_Q$ of Role B's analysis phase—part of role isolation is concentration-domain isolation.

### 8.3 Risks of the OS Analogy and Its Refinement

This paper uses the OS memory-management mapping table as its conceptual framework. But one imprecision of the analogy must be frankly acknowledged: the OS mechanism corresponding to the Concentration Criterion is **working-set selection within the page replacement algorithm** (deciding which pages should remain in physical memory), not the MMU's address-legality check (deciding whether an address can be accessed).

It is the Readiness Criterion that corresponds to the MMU—it ensures that only legitimate addresses (processed data) can be accessed. The independent existence of these two OS subsystems happens to prove the orthogonal relation of the two criteria: **the MMU is responsible for readiness checking ("whether it can"), and the page replacement algorithm is responsible for activation selection ("whether it should").** The complete theory of agent cognitive resource management corresponds exactly to these two independent subsystems of OS memory management—and this is not a coincidental analogy but evidence of structural isomorphism: two independent dimensions must exist in both domains, because they answer two different questions.

### 8.4 Limitations

1. **The computational precision of the concentration domain depends on the LLM's ability to judge "relevance"**: the semantic-similarity component of the current composite score is an approximation by the retrieval model, not genuine semantic reasoning. Future work may explore using the LLM itself for attention-budget decisions—similar to the Shícè paradigm's practice of delegating scheduling decisions to LLM runtime reasoning.

2. **Not yet validated on ultra-long-horizon tasks (100+ steps)**: the current experiments are conducted on 15–20-step tasks. Whether $W_Q$ management will encounter new degradation modes over longer time scales (e.g., the semantics of $Q$ itself legitimately evolving as it advances—unlike drift, this is the normal refinement of the goal itself) remains to be observed.

3. **The maintenance cost of the state file is borne by the user**: the current scheme requires the user to participate in reviewing and editing the state file. Fully autonomous concentration-domain computation is the long-term goal—when model capability grows to the point of autonomously maintaining $Q$ anchoring and the $W_Q$ boundary, Confidence Tags themselves will evolve from a user-interaction mechanism into a purely internal, metacognitive marker of the agent's own cognitive state.

## 9. Conclusion and Future Work

### 9.1 Main Conclusions

This paper completes the theoretical foundation of cognitive resource management for long-horizon task agents: establishing the Concentration Criterion as a second cognitive criterion orthogonal to the Readiness Criterion.

- **Criterion established**: Readiness governs **readiness** (whether injection is possible); the Concentration Criterion governs **activation** (whether injection should happen right now)—the two are orthogonal, mutually independent yet complementary
- **Axiomatic extension**: Axiom 3 (the Concentration Necessity Axiom) completes the HEEL axiom system; the three axioms correspond to three independent subsystems of OS memory management
- **A system of corollaries**: from Axiom 3, three engineering corollaries are derived—the non-growth of context occupancy, the necessity of goal anchoring, and the functional positioning of the state file
- **Engineering concepts**: the state-file template as the engineering instantiation of the concentration domain; Confidence Tags as the agent's honest declaration of its cognitive boundaries
- **Experimental validation**: a minimal sufficient experiment showing that forced concentration-domain anchoring alone significantly reduces the rate of goal drift

### 9.2 Future Work

1. **Fully autonomous concentration-domain computation**: after accumulating sufficient human-anchoring data, train models to autonomously perform $Q$ anchoring and $W_Q$ boundary cleanup, evolving Confidence Tags from a user-interaction mechanism into an endogenous metacognitive capability of the agent
2. **Cross-role concentration-domain inheritance**: when Role A hands a task over to Role B, which parts of $W_Q^{(A)}$ should be converted into the initial state of $W_Q^{(B)}$
3. **Concentration-domain arbitration under parallel goals**: when multiple goals are simultaneously active, how conflicts between different $W_Q$'s are to be detected and arbitrated
4. **Ultra-long-horizon validation**: verifying the long-term stability of concentration-domain management mechanisms on 100+-step tasks
