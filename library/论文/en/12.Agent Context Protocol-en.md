# Agent Context Protocol (ACP) — Design Draft

**Version** 3.0  
**Status** Draft  
**Date** 2026-05-13

---

## 1. Overview

### 1.1 Background

Mainstream large language model (LLM) APIs (OpenAI, Anthropic, etc.) use a flat `messages` array as the context carrier, forcing all information into just two fields: `role` and `content`. When applications need to inject structured signals — session state, timestamps, hierarchical memory, and the like — the only option is to "embed agreed-upon formats inside strings." This both increases the model's parsing burden and constrains information fidelity and training alignment.

### 1.2 Goals

We define a **context protocol for memory-capable agents**, one that:

- evolves from `Any` (flat text) to `Interface` (polymorphic, strongly typed fields);
- natively supports structures such as time, numeric state, hierarchical summaries, and tool results;
- retains the freedom of plain text, lowering the cost of developer onboarding;
- can be used directly by model providers for training, so that the model treats the protocol as its "native language";
- guarantees backward and forward compatibility through explicit version control.

---

## 2. Design Principles

1. **Polymorphic compatibility**  
   All key fields support two forms: **structured objects** or **plain-text strings**. Simple scenarios require no learning of new types; complex scenarios can exploit the advantages of structure.

2. **Training as protocol**  
   The protocol structure maps directly onto model input signals (embeddings, positional encodings, attention masks, etc.) rather than merely being "read" as text. Structured fields can be given dedicated processing channels.

3. **Transparent version semantics**  
   A mandatory top-level `protocol` field lets the server, the client, and the model determine the protocol version unambiguously, avoiding guesswork.

4. **Progressive structuring**  
   Experimental features are carried in pocket fields (`meta`, `extra_context`) and promoted to formal fields once they mature, without breaking the existing ecosystem.

5. **Human–machine co-readability**  
   The JSON format is friendly to developers, logging systems, and automated tools alike.

## 3. Protocol Specification

### 3.1 Top-Level Request

```typescript
interface ContextRequest {
  protocol: string;            // protocol version, e.g. "3.0"; required
  identity: Identity;
  engine?: Engine;             // optional; snapshot of the agent's own capabilities
  state: State;
  memory: Memory;
  current_message: CurrentMessage;
  meta?: Record<string, unknown> | string;
  extra_context?: string;
}
```

- `protocol` explicitly identifies the version, ensuring that the model or gateway can switch parsing logic as needed. A missing or unknown version should cause the request to be rejected or downgraded to plain-text processing.
- `meta` accepts arbitrary key–value pairs or a string, and is used to convey not-yet-finalized metadata.
- `extra_context` is a fully free-form textual catch-all, to which any prompting information can be appended.

### 3.2 Polymorphic Field Definitions

Every field below has the type `ConcreteStruct | string`. **When passed as a string, the model treats it as plain natural-language context; when passed as a structured object, the model can exploit the fine-grained semantics inside the field.**

#### 3.2.1 `identity` — Identity and System Instructions

**String form**  
The complete system-prompt text, compatible with existing `system` messages.

**Structured form**

```typescript
interface IdentityStruct {
  name: string;
  persona: string;          // core persona description
  tone?: string;            // tone and style
  rules?: string[];         // explicit list of constraints
  examples?: string[];      // few-shot examples
}
```

#### 3.2.2 `engine` — Engine and Tool Configuration (Optional)

**Semantic note**: `engine` represents **a snapshot of the agent's own capabilities as perceived within the current context**; it does not define or enforce tool invocation at the API level. Actual tool calls remain governed by the API's native parameters; the fields here serve only the model's metacognition, and the two should be kept as consistent as possible.

**String form**  
A natural-language description of the current model's capabilities and available tools.

**Structured form**

```typescript
interface EngineStruct {
  current_model: string;
  available_models?: string[];
  capabilities?: string[];          // e.g. "vision", "code_interpreter"
  tools?: ToolDefinition[];         // tool snapshot; should stay in sync with the API tool definitions
}

interface ToolDefinition {
  name: string;
  description: string;
  parameters: Record<string, unknown>;
}
```

#### 3.2.3 `state` — Session State

**String form**  
An informal description of current consumption, budget, mood, etc., e.g. "3,400 consumed, 6,600 remaining; please keep the reply concise."

**Structured form**

```typescript
interface StateStruct {
  // Standard state fields (a key focus of training alignment)
  last_turn_cost: number;
  accumulated_cost: number;
  remaining_budget?: number;
  session_duration_sec?: number;

  // Extension state fields (arbitrary numeric values are allowed, but keys never seen
  // in training may not be fully utilized by the model)
  [key: string]: number | undefined;
}
```

> **Note**: Extension values (e.g. `temperature`, `user_heart_rate`) can be generalized through continued training or adapters, but one should not assume the model inherently understands their semantics. New state should preferably be passed via `meta` first and promoted to a standard field once mature.

#### 3.2.4 `memory` — Hierarchical Memory

**String form**  
A text blending a summary with dialogue history, suitable for quick integration or untrained models.

**Structured form**

```typescript
interface MemoryStruct {
  summary: SummaryNode;               // hierarchical summary
  recent_dialogues: DialogueTurn[];   // slice of recent dialogue turns
}

interface SummaryNode {
  l1?: string;   // tersest summary
  l2?: string;   // more detailed summary (preferred when present)
  l3?: string;   // further levels may be added
  [level: string]: string | undefined;
}

interface DialogueTurn {
  role: "user" | "assistant" | "tool";
  content: string;
  timestamp: string;          // ISO 8601 format
  tool_calls?: ToolCall[];    // tool calls issued by the assistant
}

interface ToolCall {
  id: string;
  name: string;
  arguments: Record<string, unknown>;
  result?: unknown;           // tool return value (may appear in historical turns)
}
```

#### 3.2.5 `current_message` — Current-Turn Input

**String form**  
Simply the user's question text; treated as `role: "user"` by default.

**Structured form**

```typescript
interface CurrentMessageStruct {
  role?: "user" | "tool";             // defaults to "user"
  content: string;
  timestamp: string;
  tool_call_id?: string;              // required when role is "tool"
  attachments?: Attachment[];
}

interface Attachment {
  type: "image" | "file";
  url?: string;
  base64?: string;
  mime_type?: string;
  file_name?: string;
}
```

- For a `tool` message, `content` carries the tool execution result, and `tool_call_id` links to the corresponding `ToolCall.id`. This lets a tool result serve directly as the current-turn input, instead of being relegated to historical memory.

## 4. Request Examples

### 4.1 Pure String Form (Zero Learning Cost)

```json
{
  "protocol": "3.0",
  "identity": "You are Little Assistant; answer warmly and concisely.",
  "state": "Usage is approaching the limit; please keep subsequent replies short.",
  "memory": "The user previously asked about the weather, and you replied that it would rain.",
  "current_message": "So should I bring an umbrella?"
}
```

### 4.2 Pure Structured Form (Training-Optimized)

```json
{
  "protocol": "3.0",
  "identity": {
    "name": "Little Assistant",
    "persona": "A warm and concise personal assistant",
    "rules": ["When the remaining budget falls below 20%, reply in no more than 20 characters"]
  },
  "state": {
    "last_turn_cost": 120,
    "accumulated_cost": 3400,
    "remaining_budget": 6600
  },
  "memory": {
    "summary": { "l2": "The user asked about the weather; the assistant forecast rain." },
    "recent_dialogues": [
      { "role": "user", "content": "How's the weather today?", "timestamp": "2026-05-13T09:00:00Z" },
      { "role": "assistant", "content": "It may rain this afternoon.", "timestamp": "2026-05-13T09:00:05Z" }
    ]
  },
  "current_message": {
    "content": "So should I bring an umbrella?",
    "timestamp": "2026-05-13T10:00:00Z"
  }
}
```

### 4.3 Mixed Form (Common in Practice)

```json
{
  "protocol": "3.0",
  "identity": "You are a seasoned assistant; answer concisely.",
  "state": {
    "accumulated_cost": 5000,
    "remaining_budget": 5000
  },
  "memory": "History summary: the user is planning a trip and is interested in India's Holi festival.",
  "current_message": {
    "content": "Recommend a city.",
    "timestamp": "2026-05-13T10:15:00Z"
  },
  "meta": { "client": "web", "version": "2.1" }
}
```

### 4.4 Tool Result as the Current Input

```json
{
  "protocol": "3.0",
  "identity": "You are a weather assistant; give advice based on query results.",
  "state": "Budget is ample; reply normally.",
  "memory": {
    "recent_dialogues": [
      { "role": "user", "content": "How's the weather today?", "timestamp": "2026-05-13T11:00:00Z" },
      { "role": "assistant", "content": "Let me check.", "timestamp": "2026-05-13T11:00:01Z",
        "tool_calls": [{"id":"call_1","name":"get_weather","arguments":{"city":"Shanghai"}}] }
    ]
  },
  "current_message": {
    "role": "tool",
    "content": "{\"weather\":\"light rain\", \"temp\":20}",
    "timestamp": "2026-05-13T11:00:02Z",
    "tool_call_id": "call_1"
  }
}
```

## 5. Comparison with Existing Standards

| Dimension | OpenAI/Anthropic flat message array | This protocol, v3 |
| ------------ | ----------------------------------- | ------------------------------------------- |
| Version control | None | Mandatory top-level `protocol` field |
| Information encapsulation | Everything crammed into the `content` string | Polymorphic fields: object or string |
| Metadata support | None; requires text encoding | Native numbers, timestamps, hierarchical summaries |
| Time awareness | Relies on message order | Explicit `timestamp`, exploitable in training |
| Role distinction | `role` enum | `DialogueTurn.role` + `current_message.role` |
| Tool-result carriage | Must be passed as a separate message in the array | `current_message` natively supports the `tool` role |
| Memory management | Manual truncation/splicing required | Summary Pyramid + timestamped turn array |
| Training alignment | Model infers structure from text | Fields can map directly to dedicated training signals |
| Onboarding difficulty | Extremely low | Extremely low (string form); enhanced use requires learning the types |

## 6. Model Training Integration Guide

To truly realize "training as protocol," model providers are advised to convert the protocol structure into training sequences as follows.

### 6.1 Input Serialization Scheme

Wrap each top-level field with special tokens, so that the model can recognize field boundaries from positional signals rather than guessing from text patterns alone.

```
<|identity_start|> [content] <|identity_end|>
<|engine_start|> [content] <|engine_end|>
<|state_start|> [content] <|state_end|>
<|memory_start|> [content] <|memory_end|>
<|current_message_start|> [content] <|current_message_end|>
```

- **When a field is a string**, the content is that string; the model still acquires the information by reading text, but the field boundaries are explicit.
- **When a field is a structured object**, it is advisable to serialize it into a key–value format, for example:  
  `name: Little Assistant; persona: a warm and concise assistant; rules: [reply in no more than 20 characters when the remaining budget falls below 20%]`  
  Alternatively, use JSON lines directly, with intra-field markers (e.g. `<|state_field|>`) for finer subdivision. What matters is keeping the sequence consistent.

### 6.2 Training-Signal Injection

- For standard numeric fields in `state` (e.g. `remaining_budget`), a special role can be assigned in the attention mask, or the values can be added directly onto the positional encoding as extra embeddings.
- Auxiliary losses can be designed for tasks such as budget awareness and temporal ordering (e.g. predicting the next turn's `accumulated_cost`), so that the model genuinely learns to exploit numeric signals.
- The layered structure of `memory.summary` pairs well with curriculum learning: train on `l1` first, then introduce `l2`.

### 6.3 Polymorphic Compatibility Handling

In string form, the model treats the same field as an ordinary piece of text with explicit boundary markers, losing no natural-language understanding ability. Training must expose both string-form and structured samples, so that the model learns that "the two expressions of the same field are equivalent."

## 7. Security and Privacy Considerations

- **Transport security**: structured fields (e.g. budget, identity details) are easier to parse than plain text; all communication must be encrypted with TLS.
- **Data minimization**: clients should avoid directly exposing sensitive personally identifiable information (PII) or financial data in `state` or `meta`; apply de-identification or hashing where necessary.
- **Storage and logging**: when the server logs requests, note that the user dialogue history in `memory.recent_dialogues` may constitute sensitive content; data-retention policies should be followed and selective erasure supported.
- **Model training data**: if real conversations are used for fine-tuning, all privacy-linking information such as `timestamp` and user identifiers must be removed or anonymized, unless explicit authorization has been obtained.
- **Tool-result security**: when `current_message` carries a tool result, it may contain data from external systems; the model should follow permission and control policies to prevent unauthorized exploitation.

---

## 8. Implementation and Evolution Path

1. **Model-side adaptation**  
   During pre-training or fine-tuning, allocate dedicated embedding channels or modulation signals for structured fields according to the serialization scheme in Section 6; string forms still go through standard text encoding.

2. **API and SDK release**  
   Provide a `POST /v3/chat` endpoint that accepts JSON conforming to `ContextRequest`. The SDK automatically validates the `protocol` field and adjusts parsing behavior by version. v2 requests can be converted automatically to v3 through an adaptation layer (e.g., injecting `"protocol":"2.0"` and filling in default fields).

3. **Ecosystem tooling**  
   - Bidirectional converter: converts between the traditional `messages` array and this protocol (`messages` → `memory.recent_dialogues` and vice versa).  
   - v2-to-v3 migration tool: automatically adds the `protocol` field and wraps a legacy plain-string `current_message` into a `{ content: "..." }` structure.

4. **Progressive upgrade**  
   New features go into `meta` or `extra_context` first. Once the use cases mature and training-data coverage is sufficient, they are promoted to formal fields in a subsequent minor version (e.g. 3.1), updating the standard `StateStruct` or adding new structs.

5. **Version compatibility policy**  
   - Clients must send `protocol`.  
   - The server decides how to parse based on `protocol`: requests with an unknown major version should be rejected; within a supported major version, the server should accept all known fields and ignore unknown ones (i.e., remain forward-compatible).  
   - The model itself only needs to handle the protocol version on which it was trained; conversion can be completed at the gateway layer.

---

## 9. Advantages

- **Zero-barrier onboarding**: the pure-string form is fully compatible with existing prompt-engineering practice; developers can go live in one minute.
- **High-fidelity training**: structured fields make numbers, time, and roles first-class citizens of the model, markedly improving long-dialogue consistency, temporal reasoning, and budget awareness.
- **Exceptional extensibility**: polymorphic design + pocket fields + version control allow the protocol to evolve continuously without breaking old clients.
- **A closed loop of roles and tools**: `current_message` natively supports tool results, eliminating the perception gap; historical dialogues retain their native `role` and timestamps.
- **Training-friendly**: a clear serialization scheme and training-signal design lower integration costs for model providers.
- **An evolvable security model**: security boundaries are surfaced at the protocol level, adapting to increasingly stringent privacy regulations.
- **A differentiation moat**: once a model provider has trained this protocol into the model as its native interface, competitors cannot replicate the full capability through simple prompt engineering alone.

---

## 10. Conclusion

Building on v2's paradigm of "flat message list → polymorphic context interface," protocol v3 completes the key missing pieces: version control, native tool flow, training serialization, and security/privacy. It is not merely an API format, but **a trainable, evolvable context description language for memory-capable agents**.

Through the explicit `protocol` field and clear polymorphic field semantics, v3 strikes a further balance between extremely low onboarding cost and extremely high training optimization. As model training continues and ecosystem tooling matures, this draft has the potential to become the foundational communication standard for the next generation of strong-memory agents.
