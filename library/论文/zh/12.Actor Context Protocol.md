# 智能体上下文传输协议（ACP）设计草案

**版本** 3.0  
**状态** 草案  
**日期** 2026-05-13

---

## 1. 概述

### 1.1 背景

主流大语言模型 API（OpenAI、Anthropic 等）采用扁平的 `messages` 数组作为上下文载体，所有信息强制压缩进 `role` 和 `content` 两个字段。当应用需要注入会话状态、时间戳、分层记忆等结构化信号时，只能通过“在字符串内嵌入约定格式”来间接实现，既增加模型解析负担，又制约信息保真度与训练对齐。

### 1.2 目标

定义一套**面向记忆型智能体的上下文传输协议**，使其：

- 从 `Any`（扁平文本）进化为 `Interface`（多态强类型字段）；
- 原生支持时间、数值状态、分层摘要、工具结果等结构；
- 同时保留纯文本的自由度，降低开发者接入成本；
- 可由模型厂商直接用于训练，使模型将此协议作为“母语”；
- 通过显式版本控制保障前后向兼容。

---

## 2. 设计原则

1. **多态兼容**  
   关键字段一律支持**结构化对象**或**纯文本字符串**两种形态。简单场景无需学习新类型，复杂场景可发挥结构化优势。

2. **训练即协议**  
   协议结构直接映射为模型输入信号（嵌入、位置编码、注意力掩码等），而非仅作为文本被“阅读”。结构化字段可获得专用处理通道。

3. **版本语义透明**  
   顶层必填 `protocol` 字段使服务端、客户端和模型能明确解析协议版本，避免推断歧义。

4. **渐进结构化**  
   通过口袋字段（`meta`、`extra_context`）承载实验性特征，成熟后再升级为正式字段，不破坏已有生态。

5. **人机共读**  
   JSON 格式同时对开发者、日志系统和自动化工具友好。

## 3. 协议规范

### 3.1 顶层请求

```typescript
interface ContextRequest {
  protocol: string;            // 协议版本，如 "3.0"，必填
  identity: Identity;
  engine?: Engine;             // 可选，智能体自身能力快照
  state: State;
  memory: Memory;
  current_message: CurrentMessage;
  meta?: Record<string, unknown> | string;
  extra_context?: string;
}
```

- `protocol` 显式标识版本，确保模型或网关能按需切换解析逻辑。缺失或未知版本应拒绝请求或降级为纯文本处理。
- `meta` 接受任意键值对或字符串，用于传递未定型元数据。
- `extra_context` 为完全自由的文本兜底，可追加任何提示信息。

### 3.2 多态字段定义

以下每个字段均为 `具体结构 | string` 类型。**当以字符串形态传递时，模型将其视为纯自然语言上下文；当以结构化对象传递时，模型可利用字段内部的细粒度语义。**

#### 3.2.1 `identity` —— 身份与系统指令

**字符串形态**  
完整的系统提示文本，与现有 `system` 消息兼容。

**结构化形态**

```typescript
interface IdentityStruct {
  name: string;
  persona: string;          // 人设核心描述
  tone?: string;            // 语气、风格
  rules?: string[];         // 明确的约束列表
  examples?: string[];      // Few-shot 示例
}
```

#### 3.2.2 `engine` —— 引擎与工具配置（可选）

**语义说明**：`engine` 表示**当前上下文中智能体可感知的自身能力快照**，不是对 API 层面工具调用的强制执行定义。实际工具调用仍由 API 原生参数控制；此处字段仅供模型元认知使用，两者应尽量保持一致。

**字符串形态**  
自然语言描述当前模型能力和可用工具。

**结构化形态**

```typescript
interface EngineStruct {
  current_model: string;
  available_models?: string[];
  capabilities?: string[];          // 如 "vision", "code_interpreter"
  tools?: ToolDefinition[];         // 工具快照，应与 API 工具定义同步
}

interface ToolDefinition {
  name: string;
  description: string;
  parameters: Record<string, unknown>;
}
```

#### 3.2.3 `state` —— 会话状态

**字符串形态**  
口头描述当前消耗、预算、情绪等，如“已消耗 3400，剩余 6600，请精简回复”。

**结构化形态**

```typescript
interface StateStruct {
  // 标准状态字段（模型训练中重点对齐）
  last_turn_cost: number;
  accumulated_cost: number;
  remaining_budget?: number;
  session_duration_sec?: number;

  // 扩展状态字段（允许任意数值，但未在训练中出现过的键可能无法被模型充分利用）
  [key: string]: number | undefined;
}
```

> **注意**：扩展数值 (如 `temperature`, `user_heart_rate`) 可通过继续训练或适配器泛化，但不应假设模型天然理解其语义。建议新状态优先通过 `meta` 传递，待成熟再提升为标准字段。

#### 3.2.4 `memory` —— 分层记忆

**字符串形态**  
一段融合了摘要和历史对话的文本，适用于快速接入或未训练模型。

**结构化形态**

```typescript
interface MemoryStruct {
  summary: SummaryNode;               // 分层摘要
  recent_dialogues: DialogueTurn[];   // 近期对话切片
}

interface SummaryNode {
  l1?: string;   // 最简摘要
  l2?: string;   // 较详细摘要（存在时优先使用）
  l3?: string;   // 可继续扩展
  [level: string]: string | undefined;
}

interface DialogueTurn {
  role: "user" | "assistant" | "tool";
  content: string;
  timestamp: string;          // ISO 8601 格式
  tool_calls?: ToolCall[];    // assistant 发出的工具调用
}

interface ToolCall {
  id: string;
  name: string;
  arguments: Record<string, unknown>;
  result?: unknown;           // 工具返回结果（可存在于历史轮次中）
}
```

#### 3.2.5 `current_message` —— 当前本轮输入

**字符串形态**  
单纯的用户问题文本，默认视为 `role: "user"`。

**结构化形态**

```typescript
interface CurrentMessageStruct {
  role?: "user" | "tool";             // 默认为 "user"
  content: string;
  timestamp: string;
  tool_call_id?: string;              // 当 role 为 "tool" 时必须提供
  attachments?: Attachment[];
}

interface Attachment {
  type: "image" | "file";
  url?: string;
  base64?: string;
  mime_type?: string;
  file_name?: string;
}
```

- 若为 `tool` 消息，`content` 承载工具执行结果，`tool_call_id` 关联对应的 `ToolCall.id`。这使工具返回结果可直接作为当前轮次输入，无需退化为历史记忆。

## 4. 请求示例

### 4.1 纯字符串形态（零学习成本）

```json
{
  "protocol": "3.0",
  "identity": "你是小助手，回答温暖简洁。",
  "state": "当前消耗已接近上限，后面请尽量缩短回复。",
  "memory": "用户之前问过天气，你回答会下雨。",
  "current_message": "那我需要带伞吗？"
}
```

### 4.2 纯结构化形态（训练优化）

```json
{
  "protocol": "3.0",
  "identity": {
    "name": "小助手",
    "persona": "温暖简洁的个人助理",
    "rules": ["当剩余预算低于20%时回复不超过20字"]
  },
  "state": {
    "last_turn_cost": 120,
    "accumulated_cost": 3400,
    "remaining_budget": 6600
  },
  "memory": {
    "summary": { "l2": "用户询问天气，助手预报会下雨。" },
    "recent_dialogues": [
      { "role": "user", "content": "今天天气怎么样？", "timestamp": "2026-05-13T09:00:00Z" },
      { "role": "assistant", "content": "下午可能下雨。", "timestamp": "2026-05-13T09:00:05Z" }
    ]
  },
  "current_message": {
    "content": "那我需要带伞吗？",
    "timestamp": "2026-05-13T10:00:00Z"
  }
}
```

### 4.3 混合形态（常见实战）

```json
{
  "protocol": "3.0",
  "identity": "你是资深助手，回答简洁。",
  "state": {
    "accumulated_cost": 5000,
    "remaining_budget": 5000
  },

    "timestamp": "2026-05-13T10:15:00Z"
  },
  "meta": { "client": "web", "version": "2.1" }
}
```

{
  "protocol": "3.0",

      { "role": "user", "content": "今天天气如何？", "timestamp": "2026-05-13T11:00:00Z" },

    "content": "{\"weather\":\"小雨\", \"temp\":20}",

  }

## 5. 与现有标准的比较

| 维度         | OpenAI/Anthropic 扁平消息数组 | 本协议 v3                                    |
| ------------ | ----------------------------- | -------------------------------------------- |
| 版本控制     | 无                            | 顶层 `protocol` 字段，必填                   |
| 信息封装     | 全部塞入 `content` 字符串     | 多态字段：对象或字符串                       |
| 元数据支持   | 无，需文本编码                | 原生数值、时间戳、分层摘要                   |
| 时间感知     | 依赖消息顺序                  | 显式 `timestamp`，可训练利用                 |
| 角色区分     | `role` 枚举                   | `DialogueTurn.role` + `current_message.role` |
| 工具结果承载 | 需作为独立消息在数组中传递    | `current_message` 原生支持 `tool` 角色       |
| 记忆管理     | 需手工截断/拼接               | 金字塔摘要 + 带时间戳的轮次数组              |
| 训练对齐     | 模型从文本中推测结构          | 字段可直接映射为专用训练信号                 |
| 接入难度     | 极低                          | 极低（字符串形态）；增强时需学习类型         |

## 6. 模型训练集成指南

为真正实现“训练即协议”，建议模型厂商按如下方式将协议结构转换为训练序列。

### 6.1 输入序列化方案

使用特殊标记（special tokens）包裹各顶层字段，使模型能够从位置信号中识别字段边界，而非仅靠文本模式猜测。

```
<|identity_start|> [内容] <|identity_end|>
<|engine_start|> [内容] <|engine_end|>
<|state_start|> [内容] <|state_end|>
<|memory_start|> [内容] <|memory_end|>
<|current_message_start|> [内容] <|current_message_end|>
```

- **当字段为字符串时**，内容即为该字符串，模型仍通过文本阅读获取信息，但字段边界明确。
- **当字段为结构化对象时**，建议将其序列化为一种键值对格式，例如：  
  `name: 小助手; persona: 温暖简洁的助理; rules: [当剩余预算低于20%时回复不超过20字]`  
  或直接使用 JSON 行，配合字段内标记（如 `<|state_field|>`）进一步细分。重要的是保持序列的一致性。

### 6.2 训练信号注入

- 对 `state` 中的标准数值字段（如 `remaining_budget`），可在注意力掩码中赋予特殊角色，或作为额外嵌入直接相加到位置编码上。
- 可针对预算感知、时间顺序等任务设计辅助损失（如预测下一轮 `accumulated_cost`），使模型真正学会利用数值信号。
- `memory.summary` 的层级结构可配合课程学习：先训练利用 `l1`，再引入 `l2`。

### 6.3 多态兼容处理

字符串形态下的同一字段，模型将其视为一段带显式边界标记的普通文本，不会损失任何自然语言理解能力。训练时需同时暴露字符串和结构化两种样本，让模型学会“同一字段的两种表达等价”。

## 7. 安全与隐私考虑

- **传输安全**：结构化字段（如预算、身份细节）比纯文本更易解析，所有通信必须使用 TLS 加密。
- **数据最小化**：客户端应避免在 `state` 或 `meta` 中直接暴露敏感个人身份信息（PII）或金融数据。必要时进行脱敏或哈希处理。
- **存储与日志**：服务端记录请求时需注意 `memory.recent_dialogues` 中的用户对话历史可能属于敏感内容，应遵循数据保留策略并支持选择性清除。
- **模型训练数据**：若使用真实对话微调，需删除或匿名化所有 `timestamp`、用户标识等隐私关联信息，除非获得明确授权。
- **工具结果安全**：`current_message` 作为工具返回时可能包含外部系统数据，模型应遵循权限和控制策略，防止越权利用。

## 8. 实施与演进路径

1. **模型侧适配**  
   在预训练或微调阶段，依据第 6 节的序列化方案为结构化字段分配专用嵌入通道或调制信号；对字符串形态仍走标准文本编码。

2. **API 与 SDK 发布**  
   提供 `POST /v3/chat` 端点，接受符合 `ContextRequest` 的 JSON。SDK 自动校验 `protocol` 字段，并根据版本调整解析行为。对 v2 请求可通过适配层自动转换为 v3（例如注入 `"protocol":"2.0"` 并补充缺省字段）。

3. **生态工具**  
   - 双向转换器：支持将传统 `messages` 数组与本协议互相转换（`messages` → `memory.recent_dialogues` 及反之）。  
   - v2 到 v3 迁移工具：自动添加 `protocol` 字段，将旧有 `current_message` 纯字符串包装为 `{ content: "..." }` 结构。

4. **渐进升级**  
   新特征优先放入 `meta` 或 `extra_context`。当用例成熟且训练数据覆盖充分后，在后续小版本（如 3.1）中提升为正式字段，并更新标准 `StateStruct` 或新增结构体。

5. **版本兼容策略**  
   - 客户端必须发送 `protocol`。  
   - 服务端根据 `protocol` 决定解析方式：未知主版本应拒绝请求；支持的主版本内，应前向兼容所有已知字段（忽略未知字段）。  
   - 模型本身只需处理其训练时对应的协议版本，转换可在网关层完成。

## 9. 优势

- **无门槛接入**：纯字符串形态完全兼容现有提示工程实践，开发者 1 分钟即可上线。
- **高保真训练**：结构化字段让数值、时间、角色成为模型的一等公民，显著提升长对话一致性、时间推理和预算感知。
- **极强扩展**：多态设计 + 口袋字段 + 版本控制，使协议能在不破坏旧客户端的情况下持续演化。
- **角色与工具闭环**：`current_message` 原生支持工具返回结果，消除感知断层；历史对话保留原生 `role` 与时间戳。
- **训练友好**：清晰序列化方案和训练信号设计，降低模型厂商集成成本。
- **可演进的安全模型**：协议层面提示安全边界，适应日益严格的隐私法规。
- **差异化壁垒**：作为模型厂商，将此协议内训为模型原生接口后，竞争对手无法通过简单提示工程复制全部能力。

---

## 10. 结论

协议 v3 在 v2 “扁平消息列表 → 多态上下文接口” 的范式基础上，补齐了版本控制、工具原生流转、训练序列化以及安全隐私等关键拼图。它不仅是 API 格式，更是**一套面向记忆型智能体的、可训练、可演进的上下文描述语言**。

通过显式 `protocol` 字段和清晰的多态字段语义，v3 在极低接入成本与极高训练优化之间取得了进一步平衡。随着模型的持续训练与生态工具的完善，这一草案有潜力成为下一代强记忆智能体的基础通信标准。
