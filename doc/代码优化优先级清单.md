# 代码优化优先级清单

> 生成时间：2026-07-07
> 分析方法：文件大小 + 代码结构审查
> 优化方向：减少冗余、数据驱动、函数合并

## 一、高优先级（优化收益 > 30%）

### 1. tool/builtin.py

| 项目 | 值 |
|------|-----|
| **当前大小** | 56KB / 1377 行 |
| **优化空间** | ~40%（约 550 行） |
| **问题类型** | 冗余模式、重复结构 |
| **预计工时** | 中等（需要重写工具注册机制） |

**主要问题：**

1. **`ToolDef` 逐个手动构造**（1019-1351行）
   - 16 个工具，每个都是 `ToolDef(name=..., description=..., parameters={...}, fn=None)`
   - `parameters` schema 大量重复（`"type": "object", "properties": {...}` 结构一致）
   - **优化方向**：用数据驱动，YAML/JSON 配置表 + 代码生成

2. **`_handle_xxx` 处理器结构相似但未抽象**
   - `_handle_update_runtime`、`_handle_update_identity`、`_handle_summarize_conversation` 等
   - 都有：`加载配置 → 修改字段 → 保存 → 返回结果` 模式
   - **优化方向**：提取通用 `update_field` / `load_save` 模式

3. **handler 注册表已存在但未被充分利用**
   - `_BUILTIN_HANDLERS` 注册了函数，但 `ToolDef` 的 `fn` 全是 `None`
   - 说明工具定义和执行是分开的，但可以更紧凑

**具体可删除/合并的内容：**

| 现状 | 优化后 |
|------|--------|
| 16 个 `ToolDef(...)` 逐个写 | 配置表 + 生成器函数 |
| `_handle_update_runtime` 单独处理每种参数类型 | 循环遍历参数 schema |
| 重复的 `from character.xxx import` | 延迟导入统一管理 |
| `_ensure_tools()` 一次性构建所有工具 | 按需构建（lazy） |

### 2. character/summarizer.py

| 项目 | 值 |
|------|-----|
| **当前大小** | 39KB / 944 行 |
| **优化空间** | ~35%（约 330 行） |
| **问题类型** | 函数耦合、未利用的重复代码 |
| **预计工时** | 较小 |

**主要问题：**

1. **`_analyze_slice` / `_guess_topic` / `_describe_slice` 三函数强耦合**
   - `_analyze_slice` 返回 `(user_turns, start_t, end_t, events)`
   - `_guess_topic` 只用 `events` 来推断主题
   - `_describe_slice` 用三者组合生成描述
   - **优化方向**：合并为一个 `describe_session` 函数

2. **`_build_conversation_text` 和 `_build_topic_text` 几乎相同**
   - 差异仅 `max_chars` 参数（12000 vs 10000）
   - **优化方向**：合为一个函数，只保留 `max_chars` 参数

3. **`_l1_ensure_summary` 反复调用**
   - 在 `l1summary_to_context_string`、`build_summary_block` 等多处调用
   - 每次都检查 `s.summary` 是否为空
   - **优化方向**：在 `L1Summary` 构造时确保，避免运行时检查

4. **`compression_log` 写入分散**
   - `_append_compression_after_save` 被多处调用
   - 可以统一在 `save_l1` 里做

**具体可删除/合并的内容：**

```python
# 现状：三个独立函数
def _analyze_slice(messages):
    # ... 50行
def _guess_topic(events):
    # ... 10行
def _describe_slice(user_turns, events, topic):
    # ... 15行

# 优化后：一个函数
def describe_session(messages):
    # 分析 + 推断 + 描述 一气呵成
    # 输出 (user_turns, start_t, end_t, topic, detail)
```

### 3. common/lifecycle.py

| 项目 | 值 |
|------|-----|
| **当前大小** | 19KB / 475 行 |
| **优化空间** | ~25%（约 120 行） |
| **问题类型** | 正则解析重复、函数过长 |
| **预计工时** | 较小 |

**主要问题：**

1. **`_get_pending_triggers` 正则解析过于精细**
   - 200+ 行处理触发器状态聚合
   - 用大量正则 `re.search(r'延迟\s*(\d+)\s*s', header)` 等
   - `system_trigger` 格式相对固定，不需要这么强的解析
   - **优化方向**：简化解析，用固定分隔符替代正则

2. **`_process_triggers` 再次解析同样的内容**
   - `_get_pending_triggers` 已经解析过一遍
   - `_process_triggers` 又解析一遍（315-343行）
   - **优化方向**：一次解析，复用结果

3. **`_build_failure_reply` 过于简单**
   - 只有错误消息，可以整合到 `_post_round` 里

## 二、中优先级（优化收益 15-30%）

### 4. schedule/shice.py

| 项目 | 值 |
|------|-----|
| **当前大小** | 17KB / 470 行 |
| **优化空间** | ~20%（约 95 行） |
| **问题类型** | 长函数、分支嵌套 |
| **预计工时** | 中等 |

**主要问题：**

1. **`_advance_or_terminate` 函数过长（60行）**
   - 有 `for` 循环 + `if` 分支 + 递归调用
   - 状态推进和任务终止混在一起
   - **优化方向**：拆分为 `_advance_to_next` 和 `_terminate_job` 两个函数

2. **`_rehydrate` 恢复逻辑重复检查**
   - 延迟 > 60s 判断在两处出现（142行 和 673行过滤）
   - **优化方向**：提取为常量 `MISSED_THRESHOLD_MS = 60_000`

3. **`JobFireContext.format_trigger` 里的占位符替换**
   - 重复检查 `{pos}`、`{total}`、`{remaining}`
   - **优化方向**：单次正则替换
