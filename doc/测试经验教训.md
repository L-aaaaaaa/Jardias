# 测试经验教训


---

## 一、核心教训

### 0. 日志优先于脚本

作为语言模型，修复bug最重要的不是TDD而是日志体系，因为日志天然就是语言。所以在进行开发和调试修复之前，需要想办法把日志设计得容易看懂（包括用户和大模型）。这个优先于你着手任何开发。
并且，日志输出作为任务完成的唯一检测标准。
1. 任务开始前就以【这个任务输出的内容应该是什么样子】来设定目标（与用户对齐目标最直观的方法）。
2. 开发或者修复完成后用【现在实际运行一下，输出日志是否符合预期】来判断任务是否完成。
3. 如果判断符合预期，向用户汇报最终符合预期的日志路径，用户会检查日志，然后真实运行复现。
以上步骤，尤其是复杂开发场景下，跳过任何一个都是绝对不允许的。

细节：print和logger不要随意混用，print是对用户展示。

### 1. 真实运行 > 隔离测试

**错误做法**：为每个模块写独立的单元测试脚本，生成假的 JSON 文件存到 `history/`。

**正确做法**：找到系统已有的真实机制，直接让它生效，验证其输出。

在 Jardias0 中，`CharacterManager.append_history()` 已经在 `app.run()` 内部被调用，会自动把对话追加到
`history/{character_id}/history.jsonl`。

**结论**：测试时永远先找系统已有的机制，不要自己造一个"看起来像"的东西。

---

### 2. 测试记录要反映真实执行过程

用户说"我只需要检查历史就能了解是否真正成功"——这意味着：

- `history.jsonl` 里的内容必须是 Jarvis **实际运行时**产生的
- 不能是测试脚本自己写入的 mock 数据
- 失败也要记录（ERROR 行），这样才能看到问题出在哪一步

---

### 3. 多组件集成的 bug 会互相掩盖

开发过程中发现了多个阻断性既有 bug：

| bug                                 | 位置                              | 影响                     |
|-------------------------------------|---------------------------------|------------------------|
| `@tool` 装饰器无法解析 `Literal` 类型        | `tools/decorators.py`           | 所有工具无法创建，`CronTool` 报错 |
| `UsageHook` 引用不存在的模块                | `common/hooks/usage.py`         | 所有对话都报错，无法运行           |
| `consolidation.py` 未 import asyncio | `common/hooks/consolidation.py` | 所有对话报错                 |

这些 bug 在单独跑 pytest 时不暴露（pytest 不走 `app.run()` 完整流程），只有在真实对话时才会暴露。

**结论**：集成测试必须在完整链路上跑，不能只看单元测试。

---

### 4. 终端报错 ≠ 功能坏，检查文件内容

错误信息：

```
UnicodeEncodeError: 'gbk' codec can't encode character '\U0001f60a'
```

这是 Windows 控制台 GBK 编码与 UTF-8 emoji 字符的冲突，**不影响功能**。emoji 正常写入了 `history.jsonl`，只是在终端打印时崩溃。

**但是**：终端报错也可能掩盖真正的 bug。第一次跑时，有多个真实错误被终端编码报错掩盖（因为崩溃发生在打印阶段，不是功能执行阶段）。

**结论**：看到任何报错，先用 `try/except` 捕获后写到日志文件，确认真实错误链，再排查是终端问题还是功能问题。

### 5. 测试脚本本身要写对，不要重复写入

第一版测试脚本在 `app.run()` 返回后又自己写了一遍 `history.jsonl`，导致重复记录。

正确做法：

- `app.run()` 内部已经写了 `history.jsonl`
- 测试脚本只需要记录**测试元数据**（哪一轮、什么用例、预期是什么）
- 不要重复调用 `append_history()`

### 6. 时策系统的 IntentParser 路径在 `__main__.py`，不在 `app.run()`

"20秒后提醒我喝水" 的完整链路是：

```
IntentParser.parse() → scheduler.add_schedule() → _on_cron_job() → cron_events.jsonl
```

这条路径**不经过 LLM**。如果想让 LLM 理解定时任务，需要在 prompt 里专门设计，不是系统自动做的。

测试时策系统必须直接测 UnifiedScheduler，不能依赖 `app.run()` 绕路。

## 二、正确的测试方式

### 场景：测试 Jarvis 对话功能

**Step 1**：写一个最小化运行脚本，直接调用 `app.run()`

```python
from app import JardiasApp, DashScopeProvider, ModelChanger
import asyncio

provider = DashScopeProvider(api_key=os.environ["DASHSCOPE_API_KEY"])
app = JardiasApp(provider=provider, workspace="./workspace")

result = await app.run(character_id="test-character", message="你好")
print(result["final_content"])
```

**Step 2**：`app.run()` 内部已经调用了 `character_manager.append_history()`，直接检查生成的 `history.jsonl`

```python
# 对话结束后
with open(f"history/{character_id}/history.jsonl", encoding="utf-8") as f:
    for line in f:
        entry = json.loads(line)
        print(entry["role"], entry["content"][:80])
```

**Step 3**：不要自己在测试脚本里写 `history.jsonl`，那不是系统真实运行产生的记录。

### 场景：测试定时任务（IntentParser → Schedule → 触发）

**错误做法**：

```python
# 这是隔离测试，不是真实运行
scheduler = UnifiedScheduler(tmp)
# ...
```

**错误做法（部分）**：

```python
# app.run() 结束后 event loop 结束，cron 触发器没有机会运行
result = await app.run(character_id="cli", message="5秒后提醒我喝水")
# app.run() 返回时，5秒可能还没到；即使到了，timer 也在已结束的 event loop 里
```

**正确做法**：用 subprocess 启动 `__main__.py` CLI，保持 event loop 持续运行，等待 cron 触发，检查 stdout 和
`history.jsonl`：

```python
import subprocess, threading, time

proc = subprocess.Popen(
    [python, "__main__.py"],
    stdin=subprocess.PIPE, stdout=subprocess.PIPE, stderr=subprocess.STDOUT,
    env={**os.environ, "PYTHONIOENCODING": "utf-8"},
    bufsize=0,  # 无缓冲
)

def read():
    for line in proc.stdout:
        print(line.decode("utf-8", errors="replace"), end="")

t = threading.Thread(target=read, daemon=True)
t.start()

time.sleep(2)
proc.stdin.write(b"5秒后给我发送一个时间戳\n")
proc.stdin.flush()

time.sleep(10)  # 等待 cron 触发

proc.stdin.write(b"exit\n")
proc.stdin.flush()
proc.wait()
```

## 四、教训总结

| # | 教训                                               | 避免方法                                              |
|---|--------------------------------------------------|---------------------------------------------------|
| 1 | 不要生成假记录冒充真实运行                                    | 直接使用系统已有机制                                        |
| 2 | 测试脚本不要重复写入                                       | `app.run()` 内部已写，脚本只记录元数据                         |
| 3 | 集成链路必须真实跑                                        | pytest 通过 ≠ 系统可跑通                                 |
| 4 | 多个 bug 会互相掩盖                                     | 逐个修复，直到完整链路可跑通                                    |
| 5 | 终端报错 ≠ 功能坏                                       | 用 try/except 捕获写到日志，确认真实错误                        |
| 6 | 时策 IntentParser 路径在 `__main__.py`，不在 `app.run()` | 直接测 UnifiedScheduler，不要绕路                         |
| 7 | 测试完后把真实记录位置告诉用户                                  | 用户可以直接检查，这是可信度最高的证据                               |
| 8 | **cron 定时任务测试必须用持续运行的 event loop**               | `app.run()` 返回后 timer 就失效了；用 CLI subprocess 或手动测试 |
| 9 | 测试必须验证 `-- done:` 输出才算成功                         | 没有看到完整触发输出就说明链路还没通                                |
