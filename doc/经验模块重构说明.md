#### 触发原因：

初始化：创建角色（写012），ipu更改（写0）

## 阶段 1：IO 层抽接口（最小改动）

**目标**：`reader.py` / `writer.py` 暴露"按块"的 API，**不引入新模块**，**不改变行为**。

### 1.1 抽 reader 的 4 个块接口

```python
# experience/reader.py（新增，保持 load_experience 作为薄壳）
def read_block0(character_name) -> str: ...
def read_block1(character_name) -> str: ...
def read_block2(character_name) -> str: ...
def read_block3(character_name) -> str: ...
def read_all(character_name) -> dict[int, str]: ...   # = load_experience
```

**注意点**：
- `load_experience` 保留，内部调用 `read_all`，**所有调用方不动**
- `read_block3` 的 `_dump_written_len` 元数据剥离逻辑从 reader 提到这里
- 行为必须 100% 等价于 `load_experience()[i]`

### 1.2 抽 writer 的 5 个写入接口

```python
# experience/writer.py（新增）
def write_block0(character_name, content, engine_meta=None) -> None: ...
def write_block1(character_name, content) -> None: ...
def write_block2_append(character_name, recent_text) -> None:
    """追加到块2 近期对话原文（dump 用）"""

def write_block2_rewrite(character_name, summary_json_text, recent_text) -> None:
    """重写块2（archive 用）"""

def write_block3(character_name, user_input, timestamp) -> None: ...
def clear_block3(character_name) -> None: ...
```

### 1.3 把 `update_experience` 降级为薄壳

把现有 `update_experience` 的 5 个 operation 分支拆成 5 个函数：
- `update_experience('用户输入', ...)` → `write_block3(...)`
- `update_experience('dump', ...)` → `write_block2_append(...)`
- `update_experience('archive', ...)` → `write_block2_rewrite(...)` + meta 写入
- `update_experience('对话完成', ...)` → `clear_block3(...)`
- `update_experience('recall', ...)` → noop

`update_experience` 保留为兼容层，但**只在适配层用它**，触发层不再直接调。

### 1.4 验证
- 跑阶段 0 的测试，必须全通过
- 用 `snapshot_experience.py` 对比重构前后块内容，**必须 byte-for-byte 一致**

### 2.2 adapter/init.py

```python
"""触发原因：初始化"""

def on_register(character_name, config):
    """创建角色时调用：写块0/1/2 骨架"""
    from experience.writer import write_block0, write_block1, write_block2_rewrite
    write_block0(character_name, _render_block0_default(config))
    write_block1(character_name, _render_block1_default(config))
    write_block2_rewrite(
        character_name,
        summary_json_text="[]",
        recent_text="",
    )

def on_ipu_switch(character_name, engine):
    """切换引擎：写块0"""
    from experience.writer import write_block0
    write_block0(character_name, _render_block0_with_engine(character_name, engine))
```

### 2.3 adapter/conversation.py
```python
"""触发原因：正常对话"""

def on_user_input(character_name, user_input, timestamp):
    from experience.writer import write_block3
    write_block3(character_name, user_input, timestamp)

def on_inject_context(character_name) -> list[dict]:
    """读全 4 块，构建 messages"""
    from experience.reader import read_all
    from common.context import build_system_message
    blocks = read_all(character_name)
    return build_system_message(blocks)

def on_round_complete(character_name, new_messages):
    """追加块2，清空块3"""
    from experience.writer import write_block2_append, clear_block3
    recent = _render_recent(new_messages)
    write_block2_append(character_name, recent)
    clear_block3(character_name)
```

### 2.4 adapter/archive_recall.py
```python
"""触发原因：归档/召回"""

def on_archive(character_name, messages, summary_entry, visible_msgs):
    """重写块2 + 更新 meta"""
    from experience.writer import write_block2_rewrite
    from character.summarizer import load_compression_log, _covered_ranges

    # 1. 合并 summary
    existing = _load_summary_entries(character_name)
    merged = _merge_or_append(existing, summary_entry)

    # 2. 渲染 filtered recent
    covered = _covered_ranges(load_compression_log(character_name))
    visible = _filter_covered(visible_msgs, covered)
    recent = _render_recent(visible)

    write_block2_rewrite(character_name, merged, recent)

    # 3. 更新 _dump_meta.written_len
    _update_written_len(character_name, len(messages))

def on_recall(character_name, topic_id, recall_block):
    """归档到块2（通过 archive 操作的特殊形态）"""
    # 实际实现：recall 的 tool result 已经在 messages 里，
    # 下次 dump 会自动写入。无需直接写块2。
    # 这里留接口是为了调用方统一。
    pass
```

## 阶段 3：触发层迁移（零散 → 集中）

**目标**：把"散在各个文件的调用"统一收敛到触发层调用适配层。

### 3.1 现状触发点 → 目标调用

| 文件:函数 | 现状 | 目标 |
|-----------|------|------|
| `character/registry.py:register` | 直接调 `init_experience` | `adapter.init.on_register` |
| `tool/builtin_tools/config.py:switch_ipu` | 直接调 `sync_experience_system_block` | `adapter.init.on_ipu_switch` |
| `common/context.py:form_full_context` | 内联 `update_experience('用户输入')` | `adapter.conversation.on_user_input` + `on_inject_context` |
| `common/lifecycle.py:_post_round_async` | 调 `dump_experience` | `adapter.conversation.on_round_complete` |
| `tool/builtin_tools/experience.py:archive_recent_talk` | 调 `update_experience('archive')` | `adapter.archive_recall.on_archive` |

### 3.2 迁移顺序（按风险从低到高）
1. **switch_ipu**（最简单，改完不动 4 块骨架）→ `adapter.init.on_ipu_switch`
2. **archive_recent_talk**（块2 重写，最复杂，但独立）→ `adapter.archive_recall.on_archive`
3. **register**（低频）→ `adapter.init.on_register`
4. **_post_round_async**（每轮触发，回归测试必须全过）→ `adapter.conversation.on_round_complete`
5. **form_full_context**（最关键路径，最后改）→ `adapter.conversation.on_user_input` + `on_inject_context`

### 3.3 验证
每迁移一个点：
- 跑对应测试
- 用 `snapshot_experience.py` 比对
- 真实跑一个 5 轮对话，看块2 内容是否正常增长

## 阶段 5：块2 性能优化（可选）

**前提**：阶段 1-4 全部落地，测试稳定。

### 5.1 问题
当前 `write_block2_append` 是「读全块 → 拼 → 写全块」，对增长中的对话是 O(n²)。

### 5.2 方案
- 短期：块2 超过阈值（比如 50KB）就触发 `archive`，避免无限增长
- 中期：块2 拆成多个 `.md.x` 段文件，主文件只存索引
- 长期：换存储（SQLite），experience.md 只保留最新 1KB 摘要

## 风险点 & 应对

| 风险 | 影响 | 应对 |
|------|------|------|
| **块2 字符串模板拆散** | 行为不一致 | 阶段 0 拍快照，阶段 1 byte-for-byte 对比 |
| **迁移触发层遗漏** | 旧路径仍在用 | 在 `update_experience` 兼容层加 deprecation warning |
| **archive 的 summary 合并逻辑分散** | 数据错乱 | 合并/去重逻辑只允许在 `adapter.archive_recall` 出现，writer 不做合并判断 |
| **测试覆盖不足** | 重构漏 bug | 阶段 0 先写 `tests/test_experience_io.py`（纯 IO 测试），再覆盖 adapter |
| **`_dump_meta.json` 不在重构视野内** | written_len 错位 | 把它视为 writer 的"配套 IO"，跟着 writer 一起改 |
